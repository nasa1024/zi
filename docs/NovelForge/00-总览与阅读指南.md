## 执行摘要

**NovelForge 是什么。** 代号 NovelForge，是一个面向"网文 / 长篇连载"的 **AI 写作记忆 + 一致性 + 工艺引擎**。它不是一个聊天式写作助手，而是一条**可运行、可断路、可审计的生产线**：给定卷目标与逐章节拍契约，它能起草章节、在写时强制注入世界状态约束、在事后用确定性校验器兜底硬一致性、并行评估"追更力"、再经唯一的晋升闸门把新事实沉淀进只追加的真相源账本。技术栈 **Python + FastAPI**，本地优先，单一 SQLite 文件 `novel.db`（WAL），目标是**单人可开发可维护、长期可扩展**。

**相对原 require.md 草案纠正的 3 个地基级问题。** 原草案在三处地基上选错了方向，本设计逐一推翻重建：

1. **canon 不再是"LLM 重写的 markdown"，而是只追加的结构化账本。** 草案让 L3 通过 `update_story_bible` 整体重写 markdown，这会丢历史、无法审计、无法 as-of 投影、且把真相源交给概率性模型改写。本设计改为 **`facts` / `fact_revisions` 只追加表 + 状态机（canon/tentative/retconned）+ `valid_from_chapter`**；`story_bible.md` 降级为**从表确定性渲染的只读视图**；LLM 只产出结构化 `BibleChangeProposal`，永不写回 bible（详见第 3 节）。

2. **硬一致性不再退回"纯检索 + LLM 判断"，而是世界状态模拟 + 确定性校验器。** 草案靠相似度检索"记住"设定，本质是注意力联想，无法做全序比较、缺失边检测、状态机推演、算术守恒——恰恰是网文崩设定高发区。本设计新增 **World State Store 确定性硬状态层**（境界/知情者图/时间线/地理/库存/金手指/数值）+ `get_world_state(as_of_chapter=N)` as-of 投影，把一致性从事后纠错升级为**写时约束 + 事后兜底**双保险，由纯 Python validator（零 LLM）裁定（详见第 4 节）。

3. **双模式不再是两套系统，而是同一条管线在晋升闸门处分叉。** 草案隐含"人审版"和"全自动版"两条路。本设计确立**单一管线** `Plan→Recall→Draft→Check(continuity‖craft)→Revise→Gate→Commit`，所有 LLM 产出先入 staging，由**唯一的 `PromotionPolicy` 决策点**读 `config.canon_governance.mode` 决定 `enqueue_review`（human_gate）还是 `commit_canon`（auto_promote）——从模式 1 切到模式 2 是**改一行 config，不改代码、不换表**（详见第 3、7 节）。

**核心设计哲学（一句话五条）。**

- **软记忆走 RAG / 硬状态走关系表**：对/错有唯一答案的（境界、信息差、时序、地理、数值、库存、金手指、伏笔到期）一律 SQLite 表 + 确定性 validator；好/不好、像不像（文风、动机、氛围、模糊设定）才走 RAG + LLM-judge。两条通道物理分离、互不兜底。
- **canon 只追加**：真相源是只追加的结构化账本，永不物理删除，retcon/revert 都是新 append；markdown 永远是只读渲染产物。
- **双模式同管线分叉**：唯一管线、唯一闸门，差异仅由 config 驱动；`require_human_for[]` 是横跨两模式的安全底座，高风险变更即便全自动也强制人审。
- **追更力与一致性正交**：一致性是**不扣分项**（不写崩），追更力是**得分项**（让人想追更）；二者各有独立数据层与校验流水线，在 Check 阶段并行、互不阻塞。
- **成本/缓存是连写形态的生死线**：稳定前缀走 1h prompt cache 且绝不被章节号/检索结果污染；Opus 只留正文与冲突复核；全自动模式必有 token/美元/轮数 circuit breaker。

## 模式 1 vs 模式 2 对照表

二者是**同一条管线、同一个 `PromotionPolicy.decide()`、同一组表**，唯一差异是 `config.canon_governance.mode` 取值导致晋升闸门处的分叉走向不同。

| 维度 | 模式 1：AI 起草 + 人审定 canon | 模式 2：全自动小说家 |
|---|---|---|
| 定位 | 作者把关，每章审定 canon | 无人值守连写，尽量少打断 |
| `config.mode` | `human_gate` | `auto_promote`（或 `hybrid`） |
| 闸门处行为 | 所有候选 → `enqueue_review` | 低风险且证据达阈 → `commit_canon`；其余 → `enqueue_review` |
| 低风险软记忆（场景摘要/文风） | 入队，"按章一键批准低风险"快速放行 | `evidence_strength ≥ auto_promote_threshold` 时自动晋升，不打断 |
| 高风险（world_rule/power_system/character_death/foreshadow_payoff/知情者图变更） | 入队，逐条人审 | **仍强制人审**（`require_human_for[]`）——auto 模式绝不自动晋升高风险 |
| 与既有 canon 冲突的候选 | 入队人审（retcon 决断） | 入队人审（冲突一律回退人审） |
| 证据不足（< 阈值） | 入队人审 | 回退人审 |
| 硬一致性 critical（continuity_gate=block） | Revise≤N 轮后仍未解 → 入队/拒绝 | Revise≤N 轮后仍未解 → circuit breaker 暂停 + 入队（硬一致性在 auto 下同样阻断，不分叉） |
| 人介入频率 | 每章高频（审定 canon 是核心价值） | 仅高风险/冲突/证据不足时 |
| 成本/熔断 | 由作者节奏自然限速 | 必须有 `budget_per_chapter`（token/美元）+ `revise_max_rounds` circuit breaker |
| `promotion_log.policy_mode` 落账 | `human_gate` | `auto_promote` / `hybrid` |
| 落地阶段 | MVP1 起 | MVP3 起 |
| 共享不变量 | 都走 `fact_candidates(proposed)` → `PromotionPolicy` → `promotion_log`；canon 永远 append-only；高风险永远人审；confidence 永不作闸门 | 同左 |

> `hybrid` = 低风险 auto、高风险 human，等价于"auto_promote + 全量 require_human_for 命中即转人审"。`confidence` 在两种模式下都**只作排序、不作晋升判据**；晋升依据 = 可程序校验的 `evidence_strength`（出处可验，权重最高）+ 无冲突 + 非高风险。

## 统一术语表（Glossary）

所有章节必须用同一套命名。下表为权威拼写；个别章节出现的同义异名已收敛至第 10 节"数据契约与命名权威"，应以本表与第 10 节为准。

**三平面**
- **控制平面（Control Plane）** = `Orchestrator`（编排主循环）+ `Skill Registry`（技能注册/调度，与 Memory 平级的独立子系统，非记忆 L4）。
- **数据平面（Data Plane）** = `Memory Core`（L0 草稿 / L1 原子 / L2 场景）+ `World State Store`（确定性硬状态层）。
- **治理平面（Governance Plane）** = staging + 晋升闸门 + 审计。

**canon 真相源（只追加结构化账本）**
- `facts`：每个逻辑事实的当前游标（指向最新生效修订 + 当前 `status` ∈ {canon, tentative, retconned} + `valid_from_chapter`）。
- `fact_revisions`：该事实的完整 append-only 修订历史（op ∈ add/update/deprecate/retcon/revert）。
- `story_bible.md`：从表确定性渲染的**只读视图**（不可手改、不被 LLM 写回、可随时删除重渲）。

**候选与治理**
- `fact_candidates`（staging）：所有 LLM 产出的唯一入口，状态 `proposed` 起。
- `PromotionPolicy`：唯一晋升决策点，纯确定性可单测，读 config 分叉。
- `promotion_log`：晋升决策的 append-only 审计（actor/op/old/new/reason/evidence_refs/policy_mode）。
- `review_queue`：human_gate 待人审任务队列。
- （`canon_changelog` 已取消，见第 10 节 R7：内容变更流职责由 `fact_revisions` 承担，不再单设此表。）

**World State Store 表**
`entities`(canonical_name, aliases) · `power_ranks`(有序枚举 rank_order) · `character_power_log` · `knowledge_edges`(知情者图) · `timeline_events`(绝对 story_time) · `geo_locations` · `travel_edges` · `item_ownership` · `item_log` · `gimmick_rules` · `gimmick_usage_log` · `numeric_facts`(带 unit)。

**工艺层表**
`beats`(beat_type ∈ setup/turn/payoff_beat/tension_point/hook；value_start/value_end 为每 beat 极性属性，非 beat_type，见第 10 节 R9) · `chapter_cards` · `character_cards`(voice_profile / arc_stages) · `foreshadow`(planted→reinforced→misled→paid_off→overdue) · `pacing_state`(tension_curve)。

**检索与渲染**
`facts_fts`(FTS5 索引 canon facts) · `drafts_fts`(FTS5 索引 L0 草稿正文行)；二者均 unicode61 + 应用层 jieba 预分词（见第 10 节 R6） · `scene_vec`(sqlite-vec vec0，仅 L2 场景块) · `story_bible.md`(只读渲染视图)。

**主循环**
`Plan → Recall → Draft → Check(continuity ‖ craft) → Revise(≤N 轮) → Gate(PromotionPolicy) → Commit`。

**配置根**
`config.canon_governance{ mode: human_gate|auto_promote|hybrid; auto_promote_threshold; require_human_for[]; continuity_gate: block|warn; revise_max_rounds; budget_per_chapter{max_tokens, max_usd} }`。

**关键概念**
- **as-of 投影**：`get_world_state(as_of_chapter=N)`，把 `*_log` 只增账本回放到第 N 章末得到世界快照；写时注入 prompt + 事后兜底校验，共用同一套 replay/迁移逻辑。
- **evidence_strength**：可程序校验的出处强度，晋升主依据（权重最高）；与 LLM 自报的 `confidence` 严格区分。
- **缺失边检测**：知情者图/时间线/状态机里"应有而无"的边（如角色未知却表现得知情），靠看全量历史而非 top_k 检索判定。
- **silent invalidator**：把章节号/时间戳/uuid/每次变化的检索结果混进稳定前缀，导致 prompt cache 静默失效——头号成本杀手。

## 开放决策与推荐默认值（依据第 9 节）

下列为待定项的推荐默认值，可在 `config.canon_governance` 等处覆盖。原则是"先能跑、再调参"。

1. **`require_human_for` 默认清单** = `[world_rule, power_system, character_death, foreshadow_payoff, knowledge_edge_change]`（世界规则、境界/力量体系、角色死亡、伏笔回收、知情者图变更）。理由：五类都直接触及 World State 硬一致性，错一处污染后续全部生成，且不可由相似度挽救；auto 下仍强制人审。
2. **embedding** 不进 MVP0/MVP1，**推迟到 MVP2**，且仅作"查相似桥段"的可选增强（结构化 SQL + FTS5 永远是主路）；**优先本地 embedding 模型**（本地优先、零外部依赖、成本/隐私可控），仅本地效果不足才切 API；索引仅 `scene_vec`(sqlite-vec on L2)。
3. **境界体系配置形态** = **YAML 有序词表**（序号即偏序）+ 合法下降事件标注（自废修为/封印/伤重跌境写入 `character_power_log` 并被 validator 放行）。理由：单人维护最直观、可版本化，覆盖网文常见跌境桥段避免误杀。
4. **全自动抽检阈值 N = 10**：每自动生成 10 章人工抽检一次（抽检 1-2 章 + 跑全量确定性 validators）。**自动降级条件（满足任一即 auto_promote→human_gate）**：连续 2 次抽检出现 hard issue / 单次 hard issue 数超阈值；触达 token/美元/`revise_max_rounds` 上限；单章修订达上限仍未消除 hard issue。
5. **MVP0 成功判据** = **追更力优先**：5 名真实读者连续试读前 5 章，≥ 60%（≥ 3/5）愿看第 6 章即通过；工程辅助判据（端到端跑通、5 validator 单测、bible 只读渲染一致、cache 命中可验、索引可重建）必须全绿但不单独决定成败。
6. **日更粒度** = 单章（约 2000-3000 字/章），每章必须携带一个"核心爽点契约"（≥1 个 value_shift 或 payoff_beat/hook），作为 GenerationContract 必填输入交 ChapterDraftSkill。
7. **历史存量稿冷启动** = 纳入但置于 MVP3；反向抽取结果一律进 staging，绝不直接 commit 为 canon；大规模时按风险分层抽检 promote。

## 已知风险与下一步

### 跨节一致性硬冲突——已解决（收敛至第 10 节）

> 此前 9 节各自展开时存在的若干**字段/命名/编号冲突**，已全部由第 10 节《数据契约与命名权威》逐条裁定收敛到单一权威版本（schema 以第 02 节 DDL 为准、命名与契约口径以第 10 节为准，其余节一律交叉引用而非各自重定义）。下表逐条标注每个原冲突现由第 10 节哪条裁定解决；各节正文据此对齐后，照抄即可得到相互兼容的 DDL。

1. **交叉引用的节号系统全盘错位。** 原症状：各节"详见第 X 节"用各自内部假想编号，与实际文件对不上（如第 04 节把 World State 称"第 3 节"、检索称"第 5 节"、治理称"第 6 节"；第 06 节自指矛盾；第 03 节把存储/渲染称"第 5 节"）。**已解决——第 10 节 R1：** 所有交叉引用统一采用 SECTION_MAP 实际节号 01–11，禁止任何节内部假想编号。
2. **`facts_fts` 三套互不兼容的 schema。** 原症状：第 02 节建在 canon facts 上、第 06 节建在 L0 草稿行上（`tokenize='jieba'`）、第 08 节为第三版（`fact_id/entity/content` + `facts_fts_tri`），索引对象/列/分词器各异。**已解决——第 10 节 R6：** 拆为两张表——`facts_fts`（索引 canon facts，`fact_id UNINDEXED + subject_tok/predicate_tok/object_tok/detail_tok`）与 `drafts_fts`（索引 L0 草稿正文行，`draft_id/chapter_no/line_start/line_end UNINDEXED + body`），均 `tokenize="unicode61 remove_diacritics 2"` + 应用层 jieba 预分词（`tokenizer_version` 入 `meta_kv`），可选 `facts_fts_tri`(trigram) 兜底；禁 `tokenize='jieba'`；第 06 节建在草稿上的表改名 `drafts_fts`，第 08 节第三套混合定义删除。
3. **`fact_candidates` 四套 DDL 冲突。** 原症状：主键类型与状态枚举在 02/03/06/08 各不一致。**已解决——第 10 节 R3：** 主键统一 `candidate_id TEXT`（`cand_xxx`）；唯一状态机 `proposed → pending_review →（promoted | rejected | superseded）`，`promoted`=已离 staging 进 canon（终态），HOLD 停留 `proposed`；禁用 `approved/committed/canon` 作候选状态。
4. **`facts.id` 类型冲突。** 原症状：全文档为 TEXT 前缀化 ID，唯第 03 节 `facts.fact_id INTEGER` 及其外键为 INTEGER。**已解决——第 10 节 R2：** `facts.id TEXT PRIMARY KEY`（`fact_xxx`），所有指向 fact 的列均 TEXT；第 03 节全部 int→TEXT，禁 INTEGER。
5. **`require_human_for` 知情者图项拼写三版。** 原症状：`knowledge_graph_change`（01）/ `knowledge_edge_change`（03、09）/ `knowledge_edge`（08）。**已解决——第 10 节 R10：** 唯一拼写 `knowledge_edge_change`；第 01 节 `knowledge_graph_change`、第 08 节 `knowledge_edge`（含 `risk_class` Literal）均改之。
6. **`BibleChangeProposal` 字段不统一。** 原症状：02 用 `subject/predicate/object`、03 用 `fact_type/entity_id/old/new/valid_from_chapter`、08 用 `entity_canonical_name/field_path/risk_class`、06 用 `target_id/old/new`。**已解决——第 10 节 R8：** 单一权威契约定义在第 10 节（`op/target_id/entity/fact_type/old/new/reason/evidence_refs/valid_from_chapter/risk_class` + 校验规则），第 02/03/06/08 各自定义全部删除、改引用第 10 节。
7. **晋升状态命名分叉。** 原症状："已晋升"在 02/08 为 `committed`/`approved`、在 03 为 `canon`；决策枚举 03 为 {COMMIT, REVIEW} 两态、07 为 {STAGING, REVIEW, COMMIT} 三态。**已解决——第 10 节 R4+R5：** 候选"已晋升"=`promoted`、事实"是真相"=`canon`（R4 严格区分 `facts.status ∈ {canon, tentative, retconned}`）；决策统一 `Route = {COMMIT=commit_canon, REVIEW=enqueue_review, HOLD=hold_staging, REJECT=reject}`（R5，统一第 03 节原两态与第 07 节原三态）。
8. **`promotion_log` 双版本并存。** 原症状：第 02、03 节各给字段不同的两份 DDL，且与 `canon_changelog` 的边界需厘清。**已解决——第 10 节 R7：** `promotion_log` 唯一权威 DDL 放第 02 节（主键 `id TEXT`，采纳 `reverts_log_id` 等补充字段，append-only 触发器禁 UPDATE/DELETE）；`canon_changelog` 取消（不存在该表），内容变更流由 `fact_revisions` 承担；第 03 节不再重复建表、改引用第 02/10 节。
9. **`Polarity` / `BeatType` 枚举两版。** 原症状：第 02 节 `beat_type` 含 `value_shift` 六值、第 05 节为 `{setup, rising, turn, payoff, tension, hook}`，且第 05 节把 value_shift 当 beat 属性。**已解决——第 10 节 R9：** `beats.beat_type` 唯一取值域 = `{setup, turn, payoff_beat, tension_point, hook}`；`value_shift` 不是 beat_type，而是每 beat 的极性属性列 `value_start/value_end`（旧 `value_from/value_to` 改名）；第 05 节 `rising/tension→tension_point`、`payoff→payoff_beat`，第 02 节 CHECK 去掉 `value_shift`。

### 设计层面尚未覆盖的缺口（功能 gap，详见结构化结果 gaps）

- **`PromotionPolicy.detect_conflict()` / `score_evidence()` / `classify_risk()` 三个确定性函数只有签名，无实现规范**：冲突检测的具体 SQL/算法、evidence_strength 打分公式、风险分级规则未给出，而它们是双模式正确性的核心。**（已补——第 11 节核心算法补全。）**
- **`reconcile_drafted_state` 的 `as_transition` / `can_apply`（claim→状态迁移算子）未定义**：事后兜底的核心机制缺实现级规范。**（已补——第 11 节核心算法补全。）**
- **近邻去重（MVP2）只有目标无算法**：用什么相似度、阈值、合并策略未定。**（已补——第 11 节核心算法补全。）**
- **多卷/分支的 World State 隔离（MVP3）只有一句话**：分支维度如何进表、as-of 投影如何带分支键未设计。
- **并发模型未规范**：后台 PipelineManager worker 与 API 写、人工编辑三方并发下，除 `busy_timeout` 与 Workspace 乐观锁外，事务隔离/重试策略未系统化。
