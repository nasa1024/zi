# 18 - 缺口改造实施方案（基于 17 号调研报告）

> 针对调研确认的 5 个落后维度 + 2 个部分落后维度，给出结合当前代码库的具体改造设计。
> 每项含：现状代码锚点 → 改造设计（schema / config / 代码改动）→ 验收标准 → 风险与回滚。
> 所有新能力默认 **feature flag 关闭**，不影响现有 270+ 测试。

## 总览与实施顺序

| 里程碑 | 改造项 | 改动量 | 风险 | 依赖 |
|--------|--------|--------|------|------|
| M1 | ③ Autopilot 会话持久化 | 小 | 低 | 无 |
| M1 | ⑥ Prompt 前缀稳定化（吃 cache） | 小 | 低 | 无 |
| M2 | ② 分层叙事摘要记忆 | 中 | 低 | 无 |
| M2 | ⑤ ConStory 检错清单 + 中段加压 | 小 | 低 | 无 |
| M3 | ① 章节多候选 + 评分择优 | 大 | 中 | ⑥（控成本） |
| M4 | ④ volume_plan 批量预规划 | 中 | 低 | 无（与 ② 协同） |
| M5 | ⑦ 质量分门控 + 润色 pass | 中 | 中 | ①（复用评分器） |
| M5 | ⑧ 伏笔回收健康度 | 小 | 低 | 无 |

M1 两项是纯工程补强（修实际缺陷 + 立省钱），先做；M2 提升每章质量的"地基"；
M3 是单点收益最大项；M4/M5 在前面稳定后叠加。

数据库变更统一走现有迁移机制（`novelforge/db/migrations/` 的 `@register("N")` 装饰器，
当前 v7，本方案新增 **v8**，所有 DDL 幂等）。

## ① 章节多候选 + 评分择优（P0-1）

### 现状锚点
- `novelforge/skills/chapter_draft_skill.py:100` —— `ctx.llm.generate()` 单次调用产单稿，
  写 `workspace["draft_text"] / ["proposals"]`。
- `novelforge/control_plane/orchestrator.py:131` —— Stage 3 DRAFT 单稿直进 Stage 4 CHECK。
- 检查器已是现成的"免费评委"：`continuity_check_skill._run_hard_validators()`（确定性，零 LLM 成本）、
  `craft_check_skill` 的 7 项校验（①②③⑤⑦确定性，④⑥轻量 LLM）。

### 改造设计

**config（`novelforge/config.py`）**：
```python
@dataclass
class CandidateConfig:
    n_candidates: int = 1          # 1 = 关闭（现状）；建议生产 2-3
    judge_tier: str = "mid"        # LLM 评委档位
    temperature_spread: float = 0.15  # 候选间温度差，保证多样性
```
挂到 `NovelForgeConfig.candidates`。

**生成（orchestrator Stage 3 重构）**：
```
for i in range(cfg.candidates.n_candidates):
    invoke("chapter_draft")  # temperature = 1.0 - i * spread
    candidates.append({draft_text, proposals, draft_raw})
```
`chapter_draft_skill.run()` 增加从 `ctx.extra["temperature"]` 读温度（gateway.generate 已有
`temperature` 参数，`gateway.py:48`，目前没人传——零成本接通）。

**择优（新模块 `novelforge/craft/candidate_judge.py`）**，三级漏斗控成本：
1. **确定性预筛（零成本）**：字数 ≥1000、proposals 非空、beat_contract 合规——不合格直接淘汰；
   全军覆没时按字数取最长（保证总有产出）。
2. **硬校验打分（零 LLM 成本）**：对每个候选跑 `_run_hard_validators()`，
   hard block 数量为一票否决项（有 block 的候选排到无 block 候选之后）。
3. **LLM 评委（1 次 MID 调用）**：仅当 ≥2 个候选过筛时触发。pairwise 对比
   （EvolvR 证实 pairwise 优于 pointwise：Coherence +21.9%），prompt 给出章节目标 + beats，
   让评委从 钩子力度/节奏/人物声音/与目标契合 4 维选优并给 0-10 总分，输出 JSON。

胜者写回 `workspace["draft_text"]/["proposals"]`，落选稿与评分存
`workspace["candidate_report"]`，由 `_complete_pipeline_run` 一并序列化进 pipeline_run
新列 `detail_json`（见 v8 迁移）。Stage 4 起的流程完全不变。

**SSE/前端**：`progress_cb("candidates", "ok", {"n": 3, "scores": [7.5, 6.8, 8.2], "winner": 2})`；
Studio 进度条 `stageOrder` 加 `candidates`；human_gate 模式下二期可做"3 选 1"卡片
（彩云小梦模式），MVP 先自动择优 + 展示落选稿摘要。

**预算**：n=3 时 draft 成本 ×3 + 1 次 MID 评委。现有 `BudgetLedger`/熔断天然兜底；
建议配合 ⑥ 的 cache（3 个候选共享同一稳定前缀，DeepSeek 第 2/3 次调用前缀全命中，
实际增量成本远小于 ×3）。

### 验收
- `n_candidates=1` 时行为与现状逐字节一致（默认回归）。
- `n_candidates=3` + FakeProvider：3 次 draft 调用、胜者非空、candidate_report 落库。
- 单测：预筛淘汰逻辑、hard block 否决、评委 JSON 解析容错（复用 `_try_parse_proposals` 的抢救风格）。

### 风险与回滚
评委偏好偏差（单 LLM judge 已知问题）→ 确定性预筛 + hard block 否决在前，评委只在
"都合格"的候选里选，错选代价低。回滚 = 配置归 1。

## ② 分层叙事摘要记忆（P0-2）

### 现状锚点
- `novelforge/memory/recall.py:66` `gather_hard_context()` —— 召回全是实体/关系/数值硬状态 +
  FTS 关键词，**没有任何"前文剧情"叙事层**；`RecallPack.to_context_str()` 同理。
- `novelforge/control_plane/orchestrator.py:211` —— Stage 7 COMMIT：`_persist_draft` 后
  没有任何摘要动作。

### 改造设计

**v8 迁移新表**：
```sql
CREATE TABLE IF NOT EXISTS chapter_summaries (
    id          TEXT PRIMARY KEY,
    chapter     INTEGER NOT NULL UNIQUE,
    summary     TEXT NOT NULL,            -- 150~300 字：事件/人物变化/情绪落点/章末钩子
    volume_no   INTEGER,
    created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
-- 卷级滚动摘要直接加列，不开新表
ALTER TABLE volumes ADD COLUMN rolling_summary TEXT;  -- 迁移里用 column_exists 幂等
```
全局梗概存现有 `meta_kv`（key=`global_synopsis`），不加表。

**写入（orchestrator Stage 7，`_persist_draft` 成功后）**：
新增 `_persist_chapter_summary(conn, ctx, chapter, draft_text)`：
- FAST tier 单次调用（`max_tokens≈400`），prompt 模板固定："用 200 字概括本章：
  ①发生了什么 ②谁的状态/关系变了 ③章末悬念"；
- `INSERT ... ON CONFLICT(chapter) DO UPDATE`（重跑同章覆盖）；
- **失败静默**（try/except + 不影响 ChapterOutcome），与 `_persist_draft` 的容错风格一致。

**卷级 rollup**：每 5 章（`chapter % 5 == 0`）或写到卷末章时，把本卷已有章摘要串起来
让 FAST tier 压成 ≤300 字 `volumes.rolling_summary`。

**读出（recall.py）**：
- `RecallPack` 加字段 `chapter_summaries: list[dict]`、`volume_summary: str`；
- 新查询 `_fetch_recent_summaries(as_of, conn, n=5)`：`WHERE chapter<=? ORDER BY chapter DESC LIMIT ?`
  —— **`chapter<=as_of` 天然继承 as-of 防剧透语义**，与现有所有查询一致；
- `to_context_str()` 开头插入：
  `## 前情摘要（近5章）` + `## 本卷至今`——放在实体硬状态**之前**，因为这是 LLM 最缺的叙事语境。

**config**：`RecallConfig` 加 `summary_window_chapters: int = 5`、`enable_summaries: bool = True`。

### 验收
- 跑一章后 `chapter_summaries` 有行、字数 ≤400；recall 的 context_str 含"前情摘要"。
- as-of 测试：为第 10 章生成时召回不含第 ≥10 章摘要。
- FakeProvider 下摘要生成失败不阻断主流程（断言 outcome.ok 不变）。

### 风险与回滚
摘要质量差会污染上下文 → prompt 固定三问结构降低发散；`enable_summaries=False` 一键回滚。
每章增量成本 ≈1 次 FAST 短调用，可忽略。

## ③ Autopilot 会话持久化（P0-3，修实际缺陷）

### 现状锚点
- `novelforge/app/autopilot_manager.py:68` —— `self._sessions: dict` **纯内存**，
  进程重启全部丢失；`cleanup_stale_sessions()` 只能清内存里的僵尸。
- 每章本就是天然 checkpoint：`pipeline_run` 表已记录每章完成状态，
  `chapter_suggest.next_chapter_no()` 已能算出"写到哪了"。

### 改造设计

**v8 迁移新表（项目库内）**：
```sql
CREATE TABLE IF NOT EXISTS autopilot_sessions (
    session_id    TEXT PRIMARY KEY,
    project_id    TEXT NOT NULL,
    from_chapter  INTEGER NOT NULL,
    to_chapter    INTEGER NOT NULL,
    current_chapter INTEGER NOT NULL,
    status        TEXT NOT NULL,        -- running/degraded/.../interrupted
    policy_mode   TEXT NOT NULL,
    chapters_done INTEGER NOT NULL DEFAULT 0,
    budget_tokens_total INTEGER NOT NULL DEFAULT 0,
    budget_usd_total    REAL NOT NULL DEFAULT 0,
    consecutive_hard_issues INTEGER NOT NULL DEFAULT 0,
    last_error    TEXT,
    req_json      TEXT NOT NULL,        -- 完整启动参数（chapter_goals/预算/降级阈值）
    started_at    TEXT NOT NULL,
    finished_at   TEXT,
    heartbeat_at  TEXT NOT NULL
);
```

**写穿（autopilot_manager）**：
- `start()` INSERT；`_run_loop` 每章完成的 `with self._lock` 块内 UPDATE
  （`current_chapter/chapters_done/budget_*/heartbeat_at`）；终态（completed/error/canceled/
  circuit_broken）UPDATE status+finished_at。复用 `db.write.with_retry`。
- 状态机加一个新值 `interrupted`：**启动恢复策略选"标记不自动续跑"**——
  写小说成本敏感，重启后悄悄烧钱不可接受；由用户显式恢复。

**恢复路径**：
- `list_for_project()` 改为 内存会话 ∪ DB 会话（DB 中 running/degraded 且心跳超时
  → 现场标 `interrupted` 再返回）；
- 新端点 `POST /{project_id}/autopilot/{session_id}/resume`：读 `req_json`，
  以 `max(current_chapter, next_chapter_no())` 为新起点调 `mgr.start()`（沿用原 session_id 或
  新 id + `resumed_from` 字段，取简单的后者）；幂等：已完成的章不会重写
  （`from_chapter` 用 next_chapter_no 兜底）。
- **前端**：Studio 现有"找回会话"useEffect（`Studio.tsx` autopilot 区块）天然兼容——
  status=interrupted 的会话显示「▶ 恢复」按钮，调 resume 接口。

### 验收
- 启动 autopilot → kill 进程 → 重启 → status 接口返回 interrupted 会话 → resume 从断点继续。
- 单测：写穿后 DB 行与内存对象一致；resume 不重写已完成章。

### 风险与回滚
写穿频率 = 每章 1 次 UPDATE，无性能问题。表独立、不动现有逻辑，风险极低。

## ④ volume_plan 批量预规划（P1-4）

### 现状锚点
- 表结构全齐没人填：`chapter_cards`（schema.sql:346，goal/hook_text/summary）、
  `beats`（:324，status='planned'）、`volumes`（:583，synopsis）。
- 消费端已就绪：`novelforge/app/chapter_suggest.py` 按优先级读 chapter_cards/beats，
  `planner_skill` 也读 beats——**填上数据，"下一章自动选最优"立刻从兜底拼装升级为大纲驱动**。

### 改造设计

**新 skill `novelforge/skills/volume_plan_skill.py`**（注册进 `skills/__init__.py`）：
- contract：trigger 新增 `SkillTrigger.VOLUME_PLAN`；tier=STRONG；
  inputs=[卷 synopsis、起止章、recall_pack（实体+伏笔+全局梗概）]；
  outputs=[chapter_cards JSON 数组]。
- prompt 输出格式借鉴 chapter_draft 的双代码块风格：每章
  `{chapter, title, goal, hook_text, beats: [{beat_type, summary, value_axis}]}`，
  并显式要求（对标笔灵的工艺字段）：每章 goal 须含**爽点/冲突**，hook_text 必填，
  到期伏笔（从 recall 注入 foreshadow due 列表）必须安排回收章。
- 解析复用 `_try_parse_proposals` 的截断抢救策略；单次最多规划 **10 章**（超出分批，防截断）。

**落库**：`INSERT OR REPLACE INTO chapter_cards(status='planned')` +
`beats(status='planned')`（注意 `UNIQUE(chapter, seq)`，重规划先 DELETE 该章 planned 行）。
**只覆盖 status='planned' 的行，已 drafted/committed 的章跳过**——保护既成事实。

**API（`novelforge/app/api/volumes.py` 扩展）**：
- `POST /{project_id}/volumes/{volume_no}/plan` `{from_chapter?, to_chapter?}` → 同步执行，
  返回生成的 cards（卷范围 ≤10 章一次跑完，无需后台会话）；
- `GET /{project_id}/chapter-cards?from=&to=` + `PATCH /{project_id}/chapter-cards/{chapter}`
  —— 人审/手改入口（human_gate 精神的延伸：规划也可审）。

**前端**：Studio 生成章节面板加「📋 卷规划」区块：选卷 → 生成 → 表格展示 cards
（可编辑 goal/hook）→ 保存。挂机连写起跑前提示"本卷还有 N 章未规划"。

### 验收
- 给 volumes 填 synopsis → plan 接口产出 ≤10 章 cards+beats 入库 →
  `/pipeline/next` 的 suggested_goal 出现 chapter_card 来源（现有测试模式可直接扩展）。
- 重规划不覆盖已完成章。

## ⑤ ConStory 19 子类检错清单 + 中段加压（P1-5）

### 现状锚点
- `continuity_check_skill.py:104` `_SOFT_SYSTEM` —— 软检查只有 3 行泛泛提示
  （性格/情感弧/未铺垫能力），输出无类型体系。
- `orchestrator.py:157` —— `cfg.max_revise_loops` 全局常量 2，与章节位置无关。

### 改造设计

**清单化 prompt（纯文本改动）**：`_SOFT_SYSTEM` 重写为 5 大类 19 子类结构化清单
（来源 ConStory-Bench, arXiv:2603.05890，已核查）：
```
1 时间线/情节逻辑：绝对时间矛盾|时长冲突|同步悖论|无因之果|因果违反|废弃情节线
2 人物：记忆矛盾|知识不一致|能力波动|遗忘特技
3 世界/场景：世界规则违反|社会规范违反|地理矛盾
4 事实细节：外貌不符|命名混淆|数量偏差
5 叙事风格：视角混乱|基调不一致|风格漂移
```
输出格式升级为 `{"type":"soft","subclass":"2.3-能力波动","severity":"warn|block","desc","span"}`
——subclass 入 issue dict 后**自动**流入现有 revise prompt（`orchestrator._revise` 直接
拼 `i.get('desc')`，无需改）和前端 issue 展示。
注意：软检查目前喂 `draft_text[:3000]`（:125），清单变长后截断上限提到 6000，
max_tokens 2048 不变。

**中段加压（orchestrator）**：
- 新辅助 `_volume_progress(conn, chapter) -> float|None`：查所在卷
  `(chapter - start) / (end - start)`；无卷信息返回 None。
- Stage 5 REVISE 前：`if 0.4 <= progress <= 0.6: revise_budget = cfg.max_revise_loops + 1`；
  同时给 `progress_cb("check", ...)` 的 detail 加 `{"midpoint_boost": true}` 便于观测。
- config：`midpoint_boost: bool = True`（独立开关）。

### 验收
- 软检查输出含 subclass 字段；中段章节（构造 volumes 1-10 章，跑第 5 章）revise 上限为 3。
- 成本不变（仍是单次 MID 软检查调用）。

## ⑥ Prompt 前缀稳定化（P1-6，立省钱）

### 现状锚点（恰好是反模式）
- `chapter_draft_skill.py:86-92`：user_msg 拼装顺序 = **章节号→字数→章节目标→beats→召回**
  ——最易变的内容在最前，DeepSeek 前缀缓存（前缀完全匹配才命中）**一个字节都吃不到**。
- `gateway.generate()` 有 `cache_hint` 参数（gateway.py:49）但全代码库无人传。
- 同一章内 draft / continuity_check / craft_check / revise 共 4-6 次调用，各自拼各自的
  上下文，互相之间也无共享前缀。

### 改造设计

**统一稳定前缀块（orchestrator Stage 1 末尾构建一次）**：
```python
workspace["stable_context"] = (
    recall_pack.to_context_str_stable()   # 慢变层：taboos→gimmick→entities→canon facts
    + recall_pack.to_context_str_dynamic()  # 快变层：power/knowledge/items/numeric/timeline
)
```
`RecallPack` 拆出两个序列化方法，**慢变在前、快变在后**；所有 `_fmt_rows` 的输入
SQL 已带 ORDER BY（recall.py 各查询），保证确定性输出——这是字节级命中的前提。

**各 skill 改拼装顺序**（以 chapter_draft 为例）：
```
user_msg = stable_context            # ← 前缀，跨调用/跨候选/部分跨章命中
         + "\n\n## 本章任务\n"        # ← 动态段从此开始
         + 第N章/目标/beats
```
continuity/craft 软检查同理：`stable_context + draft_text + 检查指令`。
**收益结构**：① 同章内 4-6 次调用共享前缀（必命中）；② 多候选 ×N 次 draft 共享（①改造的
成本保险丝）；③ 跨章部分命中（taboo/entities 不变时）。DeepSeek hit 价格为 miss 的 ~2%，
README 所称"稳定前缀走 prompt cache"由此真正落地。

**观测**：`Response.usage` 透传 provider 的 cache 命中字段
（DeepSeek 返回 `prompt_cache_hit_tokens`，`openai_compat_provider.py` 加解析），
`BudgetLedger` 记 `cache_hit_tokens` 累计，pipeline done 事件带出，Studio run-summary 显示
"cache 命中率"。没有观测的优化等于没做。

### 验收
- FakeProvider 断言：同章两次 generate 的 user_msg 前 N 字节一致。
- 真实 DeepSeek 冒烟：第二个候选/check 调用的 `prompt_cache_hit_tokens > 0`。

### 风险
慢变/快变分层判断错位只损失命中率、不损正确性；纯重排无语义变化。

## ⑦ 质量分门控 + 软问题润色 pass（P2-7/8 合并）

### 现状锚点
- `orchestrator.py:156` —— revise 只救 `severity=="block"`；craft 的 warn（节奏/对话/纸片人）
  检出后**无人消费**，直接躺在 issues 列表里。
- `autopilot_manager._run_loop:209` —— 降级只数 hard issue 次数，质量崩了但不违规时不降级。
- pipeline_run 表无质量字段。

### 改造设计

**评分**：复用 ①的 LLM 评委（candidate_judge 的 0-10 评分函数单独抽出）；
n_candidates=1 时也对唯一稿打一次分（1 次 MID 调用，受 `quality.enabled` 开关控制）。
分数走 v8 迁移加列：`ALTER TABLE pipeline_run ADD COLUMN quality_score REAL` +
`detail_json TEXT`（与 ① 共用）。

**润色 pass（Stage 5 扩展）**：
```
hard_blocks 清零后：
if cfg.quality.polish_enabled and (score < cfg.quality.min_score or len(craft_warns) >= 3):
    一次性把 craft warn 列表喂 _revise()（prompt 改为"润色而非重写，保留情节与提案事实"）
    重跑 craft_check，取分数较高的版本（防润色变差——HoLLMwood 渐进精炼 + 保底）
```
计费沿用 `ledger.charge_revise_round()`，受现有 `max_revise_rounds` 熔断保护。

**门控联动**：
- `ChapterOutcome` 加 `quality_score` 字段 → autopilot `_run_loop` 中
  `score < min_score` 视同一次 hard issue 计入 `consecutive_hard_issues`
  （阈值复用 `auto_degrade_after_consecutive_issues`，连续低分 → 自动降级 human_gate）。
- API：`PipelineRunRecord` 加 `quality_score`，Studio 历史列表加分数列（趋势一目了然）。

**config**：
```python
@dataclass
class QualityConfig:
    enabled: bool = False
    min_score: float = 6.0      # autonovel 同款阈值
    polish_enabled: bool = True
```

### 验收
- 低分稿触发一次润色且最终取分高者；连续 2 章低分 autopilot 降级；
  关闭开关后零额外 LLM 调用。
