# 17 - 业界调研：AI 长篇小说生成系统对比与可借鉴优化（2026-06）

> 调研方法：5 路并行检索（开源项目 / 学术论文 / 记忆注入机制 / 多智能体与中文网文工具 / 成本与上下文工程），
> 对 8 条最关键论断做了对抗性核查（7 条 CONFIRMED，1 条部分修正）。所有结论附一手来源。

## 一、主流实现模式归纳：三代技术范式

### 第一代（2023）：顺序上下文拼接

代表：**gpt-author**（mshumer，~2.5k stars）。线性 plan→draft：生成 10 个情节候选→选最优→逐章写作，
每章把**全部前文原文**拼进 prompt，无摘要无检索。15 章约 $4、数分钟出 EPUB。
README 自承：<7 章会出问题、章节被截断；社区共识"结构完整但没有读者会读完"。
（来源：https://github.com/mshumer/gpt-author ，gpt_author_v2.ipynb 一手）

### 第二代（2024）：向量 RAG + 分层摘要

代表：**AI_NovelGenerator**（YILING0013，~5.3k stars）、**Long-Novel-GPT**（MaoXiaoYuZ）。
- AI_NovelGenerator：向量检索（k=4）+ `global_summary.txt` + `character_state.txt` + 可选 Critic-Writer 循环（Plan C）；
  局限：embedding 锁死 OpenAI 兼容接口、换模型要清库、定稿是顺序瓶颈。
- Long-Novel-GPT：**大纲→章节摘要→正文**三层自上而下扩写，正文修改自动同步大纲，多线程并行；
  README 明确全书一键生成"仍在开发"。
（来源：https://github.com/YILING0013/AI_NovelGenerator 、https://github.com/MaoXiaoYuZ/Long-Novel-GPT ）

### 第三代（2025-2026）：多 Agent + 结构化状态 + 质量门控

- **inkos**（Narcooo，**7.1k stars**，已核查）：10 Agent 顺序流水线（Planner→Composer→Architect→Writer→Observer→
  Reflector→Normalizer→**Auditor（37 维审计）→Reviser**）。三层状态：Zod 校验的 JSON truth files +
  Markdown 人类投影 + **SQLite 时序记忆库按需检索**。`inkos up` daemon 模式自动连写，非关键问题自动推进，
  需要人工判断时暂停；`hookAgenda` 防止情节钩子堆积。默认每章最多 1 次自动修订，修不动就转人工。
  （来源：https://github.com/Narcooo/inkos ）
- **autonovel**（NousResearch，~1.1k stars，已核查）：四阶段（Foundation→First Draft→Revision→Export），
  **分数门控**（foundation>7.5 才放行、章节>6.0 否则重写）+ **Swiss 制 Elo 章节锦标赛**头对头择优 +
  四角色读者面板 + 双人格审稿循环；全自动 20-30 小时产出 19 章 79,456 词完整小说。
  自承：pacing 分数有结构性天花板（~7）、第 4 轮后修订"按下葫芦浮起瓢"。
  （来源：https://github.com/NousResearch/autonovel/blob/master/PIPELINE.md ）

### 商业产品的共同选择：设定"软注入"，没有硬校验

- **Sudowrite Story Bible**：Braindump→Synopsis→Characters/Worldbuilding→Outline→Scenes→Draft 顺序管道；
  官方文档原文（已核查一字不差）："Scene and Prose Generation (in Draft) will **only look at explicitly
  mentioned** Characters and Worldbuilding elements" —— 没被场景显式提及的设定根本不注入。
- **novelcrafter Codex**：wiki 式条目，三档注入（Always/Detected/Never）+ 名称别名检测 + Global 优先；
  无 token 预算硬控，靠用户自律。
- **SillyTavern World Info**：关键词/正则/向量三模式触发 + 递归级联激活 + token 预算（百分比或绝对值）+
  9 种插入位置 + Sticky/Cooldown/Delay 定时效果；官方 issue 自承递归上限"难以确定性推理"（#2622）。
- **关键判断（记忆机制调研原文）**：当前主流商业工具均为"软注入"（设定文本塞 prompt 靠 LLM 自觉），
  **"从图谱到确定性约束的桥梁尚无商业实现"** —— NovelForge 的 facts 账本 + 确定性 validator 是真实差异化。
（来源：docs.sudowrite.com / novelcrafter.com/help / docs.sillytavern.app ）

### 学术界的机制图谱（按对工程的影响排序）

| 论文 | 核心机制 | 关键数字 |
|------|---------|---------|
| Re3 (EMNLP'22) | 计划注入 + **多候选 rerank + 一致性编辑** | 连贯性 +14%、前提相关 +20% |
| DOC (ACL'23) | 层级大纲树 + 判别器控制 | 连贯性 67.6% vs 45.1%；但叶节点实现率仅 58.5%，成本 $3-4k |
| RecurrentGPT (2023) | 自然语言模拟 LSTM：长期记忆落盘 + 短期记忆在 prompt | 消融：去掉任一层连贯性显著下降 |
| Agents' Room (ICLR'25, DeepMind) | 规划 agent 与写作 agent 解耦 + shared scratchpad | 专家评估全面偏好 |
| HoLLMwood (EMNLP'24) | Writer/Editor/Actor 角色扮演，**渐进精炼不丢稿** | 四维显著优于基线 |
| StoryWriter (CIKM'25) | 事件图大纲 + **按当前事件动态压缩历史** | 平均 8000 词 |
| DOME (NAACL'25，已核查) | **时序知识图谱**（实体-关系-实体-章节号四元组）+ 生成前冲突预检 | 显著优于 SOTA |
| ConStory-Bench (ACL'26, 微软，已核查) | 长故事一致性错误的系统分类：**5 大类 19 子类** | 错误集中在叙事进程 **40-60%** 处；错误段 token 熵高 **12-19%** |
| Self-Refine (NeurIPS'23) | 同一 LLM 生成→自评→修订 | 7 任务平均 +20%，创意类有效、推理类弱 |
| WebNovelBench (2025) | 4000+ 中文网文基准，LLM-judge 8 维评分→人类作者百分位 | LLM 已超网文均值、低于优秀作者 |

### 中文网文工具与连载实践

- **笔灵 AI**：唯一在大纲层显式暴露**审稿亮点/爽点/目标/钩子**结构字段的商业工具（黄金三章 + 爽点地图）。
- **彩云小梦**：每次续写默认给 **3 条候选走向**让用户选（用户侧 best-of-3），五种风格一键切换。
- **阅文妙笔**：官方自承"AI 无法把控网文的节奏和读者的爽点"，定位辅助工具。
- **连载崩坏实证**：纯自动连载（无章纲/人物档案/设定）通常 **5-10 章内 OOC**；喂章纲+人工改可撑约 14 天、
  完稿率 ~23%。番茄小说 2025/5 单月拒 11 万本低质 AI 书；起点全面禁 AI 正文。
  → **结论：行业证据强烈支持"结构化状态 + 校验 + 人审闸门"路线，即 NovelForge 的地基方向是对的。**
（来源：ibiling.cn/xiaoshuo、zhuanlan.zhihu.com/p/644773280、m.thepaper.cn/newsDetail_forward_31741852 等）

### 成本与上下文工程共识

- **DeepSeek context caching**（官方文档）：全自动、最小缓存单元 64 tokens、前缀完全匹配才命中；
  cache hit 比 miss 便宜约 98%+（v4-flash $0.0028/M vs $0.14/M）。Anthropic 则需手动 `cache_control`，
  读取 0.1×，写入 1.25×/2×。→ **prompt 结构必须"稳定前缀在前、动态内容在后"才吃得到折扣。**
- **长上下文 vs RAG**（Li et al., EMNLP'24）：资源充足时长上下文平均质量更高，但 RAG 成本低 2-3 个数量级；
  Self-Route（按需路由）是折中。Lost in the Middle：相关信息在上下文中部时性能显著下降（U 形曲线）。
- **RAG 防剧透**：语义检索会把角色结局段落检回来答"初始动机"问题（spoiler leakage，arXiv 2512.07474）——
  **必须按叙事时序过滤**。NovelForge 的 as-of 投影天然规避此坑，是领先点。
- **断点恢复**：LangGraph checkpoint 模式（每节点落盘状态快照、可跨机器数月后恢复）是长流水线事实标准。

## 二、与 NovelForge 逐维度对比

| 维度 | 业界主流做法 | NovelForge 现状 | 差距判断 |
|------|------------|----------------|---------|
| 设定存储 | 软注入（文本条目塞 prompt）：Story Bible / Codex / Lorebook | **只追加 facts 账本 + as-of 投影 + 确定性 validator** | **领先**（商业界无硬校验实现） |
| 时序防剧透 | 标准 RAG 有 spoiler leakage，需特殊设计 | as-of 投影原生解决 | **领先** |
| 叙事记忆 | 分层摘要（场景→章→全局）+ 动态压缩（StoryWriter）+ 长短期双层（RecurrentGPT） | 实体/关系召回为主，**无前文叙事摘要层** | **落后**：硬状态强、软记忆弱 |
| 大纲驱动 | 大纲→细纲→正文三层（Long-Novel-GPT/DOC/Sudowrite），先规划后写 | 有 chapter_cards/beats/volumes 表但**无批量预规划生成**，目前逐章即时 plan | **落后**：表结构在、生产流程缺 |
| 多候选择优 | Re3 rerank、gpt-author 10 选 1、彩云小梦 3 选 1、autonovel Elo 锦标赛 | **无**：单草稿直进 check | **落后**（最大单点提升机会） |
| 修订循环 | inkos 审计→修复→复审；HoLLMwood editor 渐进精炼；Self-Refine | 有 revise loop（≤2 轮）但只对 hard block 触发；craft 软问题不修 | 持平偏弱 |
| 质量门控 | autonovel 分数阈值（<6 重写）；inkos 37 维审计 | gate 只管**事实晋升**，不管文学质量分 | **缺口**：质量无量化闸门 |
| 工艺控制 | 笔灵爽点/钩子字段；阅文承认 AI 控不住节奏 | foreshadow/beats/pacing_cursor/value_shift 结构化表 | **领先**（结构化程度高于多数） |
| 自动连写 | inkos daemon（人工判断点暂停）；gpt-author 全自动（质量差） | Autopilot 后台会话 + 预算熔断 + 自动降级 + 前端挂机 | 持平偏强 |
| 断点恢复 | LangGraph checkpoint 落盘、跨重启恢复 | **Autopilot 会话仅存内存，进程重启即丢** | **落后** |
| 一致性检错 | ConStory 19 子类分类；DOME 时序 KG 冲突预检 | continuity_check skill + Python validators | 持平；可借检错清单强化 |
| 成本工程 | 前缀稳定化吃 cache（hit 省 98%）；分层模型路由 | 有三档模型 tier + 预算熔断；**prompt 前缀稳定性未系统设计** | 部分落后 |
| 人审模式 | Sudowrite 全人审；inkos 关键点人审；gpt-author 全自动 | human_gate/auto_promote/hybrid 单闸门分叉 | **领先**（光谱设计更干净） |

总评：NovelForge 在**硬一致性、时序投影、工艺结构化、人审光谱**四个维度领先业界；
在**多候选择优、叙事摘要记忆、批量预规划、质量门控、会话持久化**五个维度有明确可补的缺口。

## 三、可借鉴优化清单（按优先级排序）

### P0-1 章节多候选 + 评分择优（直接补"自动选最优"的最后一环）
- **出处**：Re3 的 rerank（+14% 连贯性）、autonovel Elo 锦标赛、彩云小梦 best-of-3、EvolvR（pairwise 评分 Coherence +21.9%）。
- **落地**：draft 阶段并行生成 2-3 个候选（可用 fast tier 降成本），复用现有 continuity_check + craft_check
  对每个候选打分（craft 得分为主、一致性 hard block 一票否决），自动选最优进 gate；
  human_gate 模式下前端展示 2-3 个候选让作者点选（彩云小梦模式）。
  改动点：`orchestrator.generate_chapter` DRAFT 阶段 + `chapter_draft_skill` 加 `n_candidates` 参数 + Studio 候选卡片 UI。

### P0-2 分层叙事摘要记忆（章摘要 → 卷摘要 → 全局梗概）
- **出处**：RecurrentGPT 双层记忆（消融证实缺一不可）、StoryWriter 动态历史压缩、社区实践三级摘要。
- **落地**：每章 COMMIT 后用 fast tier 生成 150-300 字章节摘要存新表 `chapter_summaries`（含 as-of 章节号）；
  recall 时注入"最近 3-5 章摘要 + 当前卷摘要 + 全局梗概"，与现有实体/关系召回互补。
  解决当前"硬状态强、剧情语境弱"的偏科——validator 管得住等级数值，管不住"上一章的情绪余韵"。

### P0-3 Autopilot 会话持久化（断点续写）
- **出处**：LangGraph checkpoint 模式（节点级落盘、跨重启恢复）；inkos daemon 可从任意章节 ID 续跑。
- **落地**：`autopilot_manager` 把会话状态写入 SQLite 新表 `autopilot_sessions`（现仅内存，重启即丢，
  与"挂机连写"卖点直接冲突）；启动时扫描未完成会话提供"继续"入口。每章本就是天然 checkpoint。

### P1-4 批量预规划：先填章节卡，再连写
- **出处**：DOC/Long-Novel-GPT 的"大纲→细纲→正文"分层；Agents' Room 规划/写作解耦；
  连载实证"喂章纲可撑 14 天 vs 裸奔 5-10 章崩"。
- **落地**：新增 `volume_plan` skill：输入卷 synopsis，一次生成未来 N 章的 chapter_cards
  （title/goal/hook_text）+ planned beats，人审后入库。现有 `/pipeline/next` 的目标拼装立即变得有料——
  这是把"自动选最优"从兜底拼装升级为真正大纲驱动的关键一步。表结构已齐，只缺生产流程。

### P1-5 用 ConStory 19 子类清单强化 continuity_check，中段加压
- **出处**：ConStory-Bench（ACL'26，已核查）：5 大类 19 子类错误分类；错误集中在叙事进程 40-60%；
  错误段 token 熵高 12-19%。
- **落地**：把 19 子类（时间线矛盾/知识不一致/能力波动/地理矛盾/视角混乱…）写进 continuity_check 的
  prompt 作为结构化检查清单；对处于卷中段（40-60% 进度）的章节把 max_revise_loops 从 2 提到 3 或
  强制二次 check。零架构改动，纯 prompt + 配置。

### P1-6 Prompt 前缀稳定化，吃满 DeepSeek cache
- **出处**：DeepSeek 官方：前缀完全匹配才命中，hit 省 ~98%；Anthropic 同理（0.1× 读取）。
- **落地**：审计各 skill 的 prompt 拼装顺序，强制"system + canon 渲染（按卷冻结快照）+ 写作规范"为
  字节级稳定前缀，recall/章节目标等动态内容一律放后段。README 写了"成本/缓存是连写形态的生死线"，
  这条是兑现它成本最低的方式。

### P2-7 软问题润色 pass（editor 渐进精炼）
- **出处**：HoLLMwood editor 渐进修订不丢稿；Self-Refine（创意任务 +20%）；AI_NovelGenerator Plan C。
- **落地**：现有 revise 只救 hard block；增加可选的单轮"craft 润色"——craft_issues 非空且预算余量充足时，
  把 issues 列表喂给 revise prompt 做一次精修。配置开关 + 预算计费沿用现有 `charge_revise_round`。

### P2-8 章节质量分数门控 + 自动降级联动
- **出处**：autonovel 分数阈值门控（<6.0 重写）；WebNovelBench 8 维 LLM-judge；inkos 审计不过转人工。
- **落地**：craft_check 输出 0-10 总分入 `pipeline_run` 新列；Autopilot 中连续 N 章低于阈值时触发
  现有 auto_degrade 机制（目前只看 hard issue 次数，加上质量维度）。Studio 历史列表显示分数趋势。

### P2-9 伏笔回收健康度（hookAgenda 思路）
- **出处**：inkos `hookAgenda` 防 plot thread 堆积。
- **落地**：现有 foreshadow 表已有 due_chapter/overdue 状态，补一个聚合视图/接口：未回收伏笔数、
  最老逾期章数、每章新增 vs 回收比；超阈值时在 `/pipeline/next` 的建议里把"回收伏笔"置顶并在 Studio 亮黄牌。

### 不建议跟进的方向
- **DOC 式判别器控制**：精度最高但叶节点实现率仅 58.5%、成本 $3-4k 级，性价比差。
- **MCTS 大纲搜索（BiT-MCTS/SuperWriter）**：研究前沿，计算成本高，等社区有可复用实现再说。
- **向量 RAG 全面替换 BM25**：spoiler leakage + 调试不透明 + embedding 锁定（AI_NovelGenerator 的教训）；
  现有"实体/关系为主 + BM25 为辅 + as-of 过滤"更可控，最多在风格检索场景补向量。
