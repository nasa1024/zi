## 10. 数据契约与命名权威

> 本节是 NovelForge 全文档对 **9 处跨节冲突表/契约定义**的**唯一权威源（Single Source of Truth for Schema）**。凡 §02/§03/§06/§08 与本节冲突者，**一律以本节为准**；冲突节须按"命名权威表"末列改写为"引用 §10"，不得各自保留旧定义。
>
> 本节只裁定**契约形态**(DDL 列 / 枚举 CHECK / Pydantic 字段 / 拼写 / 节号引用)，不重复业务算法——晋升决策算法见 §03，validator 实现见 §04，召回排序见 §06，API 契约见 §08。

#### R2 — `facts` 主键及所有 fact 引用统一 `TEXT` 前缀化 ID

**(a) 权威定义**

`facts.id` 为 `TEXT PRIMARY KEY`,值为应用层生成的前缀化 ID(`fact_` + ULID/序号,如 `fact_000123`、`fact_pw_lin`)。**所有**指向 fact 的列(无论叫 `fact_id` / `target_fact_id` / `target_id` / `committed_fact_id` / `secret_fact_id` / `current_revision_id` 所指对象等)一律 `TEXT`,外键类型与 `facts.id` 严格一致。

```sql
-- §02 权威保持不变(facts.id 已是 TEXT);此处声明所有引用列的类型铁律:
-- facts.id                         TEXT PRIMARY KEY          -- fact_xxx
-- fact_revisions.fact_id           TEXT NOT NULL REFERENCES facts(id)
-- fact_candidates.target_fact_id   TEXT REFERENCES facts(id)
-- fact_candidates.committed_revision_id TEXT                 -- → fact_revisions.id (rev_xxx)
-- promotion_log.fact_id            TEXT REFERENCES facts(id)
-- knowledge_edges.secret_fact_id   TEXT REFERENCES facts(id)
-- power_ranks/character_power_log/item_log/...的 fact_id 列  TEXT
```

Pydantic 侧(权威):

```python
class Fact(BaseModel):
    id: str                                  # fact_xxx,前缀化 TEXT(禁 int)
    entity_id: Optional[str] = None          # ent_xxx,TEXT
    current_revision_id: str                 # rev_xxx,TEXT
    # ...其余见 §02.2.9
```

**§03 必改项**:把 §03.2 `fact_candidates` DDL 中 `candidate_id INTEGER`、`entity_id INTEGER`、`target_fact_id INTEGER`、`committed_fact_id INTEGER` 及 `promotion_log` 中 `candidate_id/target_fact_id/entity_id INTEGER`、§03.1 `BibleChangeProposal` 中 `target_id: Optional[int]` / `entity_id: Optional[int]` **全部改 `TEXT` / `Optional[str]`**(外键随之)。

**(b) 为何这样裁定**

§02 通用列约定第 1 条已明文规定"主键统一 `id TEXT PRIMARY KEY`,前缀化 ID,便于跨表引用与日志可读";§02 的 `facts.id`、`fact_revisions.fact_id`、`fact_candidates.target_fact_id` 全是 `TEXT`。唯独 §03 用了 `INTEGER PRIMARY KEY AUTOINCREMENT`,与 §02 直接冲突——同一外键两种类型会让 SQLite 隐式转换、`REFERENCES` 校验形同虚设,并破坏"日志可读"(整数 ID 无前缀无法肉眼判断是 fact 还是 candidate)。§02 是 DDL 权威节(SECTION_MAP),故收敛到 `TEXT` 前缀化。

#### R3 — `fact_candidates` 主键 + 候选状态机

**(a) 权威定义**

主键 `candidate_id TEXT`(`cand_xxx`)。候选状态机**唯一**为:

```
proposed → pending_review → (promoted | rejected | superseded)
```

- `proposed`:LLM 刚产出、未经 PromotionPolicy 裁定(唯一入口态)
- `pending_review`:已入 `review_queue`,等待人审(由 Route=REVIEW 落入)
- `promoted`:**已离开 staging 进入 canon**(终态;由 Route=COMMIT 或人审 approve 达成)
- `rejected`:被拒(终态)
- `superseded`:被后续 retcon 取代(终态)

**禁用** `approved` / `committed` / `canon` 作候选状态(候选只说"已晋升 = `promoted`",事实才说 `canon`,见 R4)。

权威 CHECK:

```sql
CREATE TABLE IF NOT EXISTS fact_candidates (
    candidate_id    TEXT PRIMARY KEY,                 -- cand_xxx (R2/R3:TEXT)
    -- ...(其余列见 §02.2.3,target_fact_id/entity_id 均 TEXT)
    status          TEXT NOT NULL DEFAULT 'proposed'
        CHECK (status IN ('proposed','pending_review','promoted','rejected','superseded')),
    committed_revision_id TEXT,                        -- 晋升后回填 fact_revisions.id
    -- ...
    FOREIGN KEY(target_fact_id) REFERENCES facts(id)
);
```

合法迁移(每条边落 `promotion_log`):

| from | to | 触发 |
|---|---|---|
| `proposed` | `promoted` | Route=COMMIT(auto 低风险无冲突) |
| `proposed` | `pending_review` | Route=REVIEW(human_gate / 高风险 / 冲突 / 证据不足) |
| `proposed` | `superseded` | 入 staging 即被更新候选取代(罕见) |
| `pending_review` | `promoted` | 人审 approve |
| `pending_review` | `rejected` | 人审 reject |
| `pending_review`/`promoted` | `superseded` | 后续 retcon |

Route=HOLD 不迁移状态:候选**留在 `proposed`** 等证据积累(见 R5)。

Pydantic 侧:

```python
class FactCandidate(BaseModel):
    candidate_id: str                                 # cand_xxx
    status: Literal["proposed","pending_review","promoted","rejected","superseded"] = "proposed"
    target_fact_id: Optional[str] = None
    committed_revision_id: Optional[str] = None
    # ...见 §02.2.9
```

**(b) 为何这样裁定**

三节给了三套不兼容状态机:§02 = `proposed/approved/rejected/committed/superseded`(`id` 为 PK)、§03 = `proposed/pending_review/canon/rejected/superseded`(`candidate_id INTEGER` PK)、§06 = `proposed/promoted/rejected`、§08 = `proposed/approved/rejected/superseded`。问题:① §03 把 `canon` 当候选状态,与事实自身状态(R4)语义重叠,治理代码无法区分"这是候选记录还是真相记录";② `approved` 与 `committed` 双态冗余(approve 后必 commit,中间态无落库价值,反而制造"approved 但未 committed"的悬空窗口);③ 主键类型分裂(R2)。裁定取 §03 的 `pending_review`(它精确表达"在 review_queue 里")+ §06 的 `promoted`(它精确表达"已离 staging 进 canon",且不与事实的 `canon` 撞词),删掉 `approved`/`committed`/`canon`。主键统一 `candidate_id TEXT` 兼顾 §03 的列名习惯与 §02 的 TEXT 铁律。

#### R4 — `facts.status`(事实表自身状态)

**(a) 权威定义**

```sql
status TEXT NOT NULL DEFAULT 'tentative'
    CHECK(status IN ('canon','tentative','retconned'))
```

语义:`tentative`(暂定,未晋升)、`canon`(已晋升生效的真相)、`retconned`(被 retcon 逻辑失效,物理保留)。

```python
class FactStatus(str, Enum):
    canon = "canon"
    tentative = "tentative"
    retconned = "retconned"
```

**与 R3 的硬边界(铭记)**:

| 概念 | 用词 | 表 |
|---|---|---|
| 候选"已晋升离开 staging" | **`promoted`** | `fact_candidates.status` |
| 事实"是当前生效真相" | **`canon`** | `facts.status` |
| 事实"被取代失效" | **`retconned`** | `facts.status` |
| 候选"被取代" | **`superseded`** | `fact_candidates.status` |

`promoted ≠ canon`,`superseded ≠ retconned`——前者描述候选记录的治理生命周期,后者描述事实记录的真相生命周期。一次成功晋升 = 候选 `promoted` **且** 新事实 `canon`(同一事务两处状态)。

**§03 必改项**:§03.2 状态机表与 mermaid 图里"`proposed → canon`"的箭头,应理解为"候选 `proposed→promoted`,同事务写入一条 `status=canon` 的 fact",不可把 `canon` 写进 `fact_candidates.status` 的 CHECK(已在 R3 删除)。§03.7 retcon 把旧 fact 置 `superseded` 的说法**改为 `retconned`**(事实失效用 `retconned`,§02 facts CHECK 无 `superseded`)。

**(b) 为何这样裁定**

§02 facts CHECK 是 `{canon, tentative, retconned}`,而 §03.7 retcon 段落写"旧 `facts.status` 由 `canon` 置为 `superseded`"——但 §02 的 facts 根本没有 `superseded` 这个合法值,该 UPDATE 会被 CHECK 拒绝。`superseded` 是候选词汇(R3),事实失效的词汇是 `retconned`。两套生命周期(候选治理 vs 事实真相)必须用两组互不重叠的词,否则"promoted 的候选对应的 fact 到底该查 status=promoted 还是 canon"会让每个查询都要猜。§02 为 DDL 权威,facts 三态不动,retcon 统一写 `retconned`。

#### R5 — PromotionPolicy 决策枚举 `Route`

**(a) 权威定义**

```python
class Route(str, Enum):
    COMMIT = "commit_canon"     # 直接写 canon:候选 →promoted,新 fact →canon
    REVIEW = "enqueue_review"   # 入 review_queue 等人审:候选 →pending_review
    HOLD   = "hold_staging"     # 留 staging 待证据积累:候选维持 proposed,不入队不写 canon
    REJECT = "reject"           # 拒:候选 →rejected
```

四态语义与副作用:

| Route | 含义 | 候选状态迁移 | 落账 op | 典型触发 |
|---|---|---|---|---|
| `COMMIT` | commit_canon | `proposed`→`promoted` | `commit_canon` | auto/hybrid,低风险 + 无冲突 + 证据达阈 |
| `REVIEW` | enqueue_review | `proposed`→`pending_review` | `enqueue_review` | human_gate / 命中 require_human_for / 高风险 / 有冲突 |
| `HOLD` | 留 staging | `proposed`(不变) | `hold_staging` | 证据不足阈值但**未矛盾**,等后续章补证据再裁(避免过早拒绝可能正确的提案) |
| `REJECT` | 拒 | `proposed`/`pending_review`→`rejected` | `reject` | 人审否决 / 确定性硬冲突无法 retcon |

`PromotionPolicy.decide()` 返回 `Route`;`promotion_log.decision` 的 CHECK 同步为四态:

```sql
-- promotion_log.decision(权威 DDL 在 §02,见 R7)
decision TEXT NOT NULL
    CHECK(decision IN ('commit_canon','enqueue_review','hold_staging','reject','revert'))
    -- revert 为单条回滚专用,不由 decide() 产出
```

**(b) 为何这样裁定**

§03.3 `Decision` 只有两态 `{COMMIT, REVIEW}`,§07 用 `{STAGING, REVIEW, COMMIT}` 三态。两者都缺"明确拒绝"(REJECT)与"留置等证据"(HOLD)的独立语义:§03 把"证据不足"也塞进 REVIEW(让人审去拒),把队列塞满本可自动留置的低证据项;§07 的 STAGING≈HOLD 但缺 REJECT。统一为四态 `{COMMIT, REVIEW, HOLD, REJECT}`:HOLD 收敛 §07 的 STAGING(证据不足但未矛盾 → 留 proposed,不打扰人),REJECT 显式化"确定性硬冲突/人审否决"。这让决策点对每种处境都有精确出口,且 `promotion_log.decision` 能 1:1 审计。`revert` 不是 decide() 的产物(它是事后回滚,§03.7),但同表枚举需容纳。

#### R6 — 全文检索拆两张表:`facts_fts` 与 `drafts_fts`

**(a) 权威定义**

两张独立 FTS5 表,职责分明;均 `unicode61` + **应用层 jieba 预分词**(写入前 `jieba.lcut` 空格 join,查询端同口径切词);`tokenizer_version` 入 `meta_kv`。可选第三张 `facts_fts_tri`(trigram)做人名/术语子串兜底。

```sql
-- (1) facts_fts:索引 canon facts 的可检索文本列(权威在 §02.8.1)
CREATE VIRTUAL TABLE IF NOT EXISTS facts_fts USING fts5(
    fact_id    UNINDEXED,        -- 回连 facts.id
    subject_tok,                 -- jieba 预分词:主体
    predicate_tok,               -- jieba 预分词:谓词
    object_tok,                  -- jieba 预分词:值
    detail_tok,                  -- jieba 预分词:detail_json 可读文本
    tokenize = "unicode61 remove_diacritics 2"
);

-- (2) drafts_fts:索引 L0 草稿正文行,用于"查相似桥段 / 原文检索"
CREATE VIRTUAL TABLE IF NOT EXISTS drafts_fts USING fts5(
    draft_id   UNINDEXED,        -- 回连 draft_index.id / drafts.draft_id
    chapter_no UNINDEXED,
    line_start UNINDEXED,        -- source_ref 行区间回指
    line_end   UNINDEXED,
    body,                        -- jieba 预分词后的草稿正文行区间文本
    tokenize = "unicode61 remove_diacritics 2"
);

-- (3) 可选:facts_fts_tri(trigram 子串兜底,人名/术语精确子串)
CREATE VIRTUAL TABLE IF NOT EXISTS facts_fts_tri USING fts5(
    fact_id UNINDEXED, content, tokenize = 'trigram'
);
```

`tokenize` 一律 `unicode61`(因输入已是空格分隔词串);**禁止** `tokenize='jieba'` 这种"FTS5 内置 jieba 分词器"写法(jieba 在应用层做,FTS5 只按空白切)。

**§06 必改项**:§06.1 `capture()` 与 DDL 中建在 L0 草稿上的 `facts_fts`(含 `draft_id/chapter_no/line_start/line_end/body` 列)**全部改名 `drafts_fts`**;§06.4.2 `recall_bm25` 中 `FROM facts_fts`(查草稿原文)改 `FROM drafts_fts`。§06 中 `tokenize='jieba'` 改为 `unicode61` + 注明应用层预分词。

**§08 必改项**:§08.5 把"`facts_fts(content)` 单表混装 + 注释含糊"那套删除,采纳本节两表拆分;`facts_fts_tri` 保留为可选兜底。

**(b) 为何这样裁定**

三套定义打架:§02.8.1 `facts_fts` 索引的是 **facts 的结构化列**(`subject_tok/predicate_tok/...`);§06.1 的 `facts_fts` 索引的却是 **L0 草稿正文行**(`body/line_start/line_end`);§08.5 又是第三套(`facts_fts(fact_id, entity, content)` + `facts_fts_tri`)。同名表三种 schema,建表必崩。本质是**两个不同检索对象**被错塞进一个名字:canon 事实检索(给 Recall 主路补充关键词)vs 草稿原文检索(给作者"查相似桥段/上次怎么写的")。必须拆名:facts 的留 `facts_fts`(§02 权威),草稿的独立成 `drafts_fts`。jieba 统一在应用层(§02.8.1 已论证 FTS5 无内置中文分词器,且词典版本需绑 `meta_kv`),故 `tokenize` 恒为 `unicode61`,§06 的 `tokenize='jieba'` 是不可实现的伪写法,删除。

#### R7 — `promotion_log` 单一权威 DDL + `canon_changelog` 取消

#### R8 — `BibleChangeProposal` 单一 Pydantic 契约

**(a) 权威定义(权威在 §10,其余节引用)**

```python
from typing import Literal, Optional
from pydantic import BaseModel, Field, model_validator, field_validator

class BibleChangeProposal(BaseModel):
    op: Literal["add", "update", "deprecate", "retcon"]
    target_id: Optional[str] = None        # add 时为 None;update/deprecate/retcon 必填,指向 facts.id (R2:TEXT)
    entity: str                            # 该 fact 归属实体的稳定标识(canonical_name 或 ent_xxx)
    fact_type: str                         # world_rule|power_system|character_trait|... (取值见 §02 facts CHECK)
    old: Optional[dict] = None             # 旧值快照(add 时 None;update/deprecate/retcon 回填,供审计与回滚)
    new: dict                              # 新值(结构化,非自然语言);deprecate 也需给终态快照
    reason: str = Field(min_length=1)      # 变更理由(供审校与审计)
    evidence_refs: list[str] = Field(default_factory=list)  # 出处:chap:42#para:7 / fact_xxx / draft 行区间,可程序校验
    valid_from_chapter: int = Field(ge=0)  # canon 绑定的生效章(故事内时序)
    risk_class: str                        # 由 classify_risk 填:world_rule|power_system|character_death|foreshadow_payoff|knowledge_edge_change|soft

    @model_validator(mode="after")
    def _op_target_consistency(self):
        if self.op in ("update", "deprecate", "retcon") and not self.target_id:
            raise ValueError(f"op={self.op} 必须提供 target_id")
        if self.op == "add" and self.target_id:
            raise ValueError("op=add 不应携带 target_id")
        return self

    @field_validator("evidence_refs")
    @classmethod
    def _need_evidence(cls, v: list[str]):
        if not v:
            raise ValueError("evidence_refs 不可为空:晋升依据=可程序校验的 evidence_strength(硬原则8)")
        return v
```

**§02/§03/§06/§08 必改项**:四节各自的 `BibleChangeProposal` 定义**全部删除**,改为"`BibleChangeProposal` 权威契约见 §10.3-R8"。具体:§02.2.9 的版本(用 `subject/predicate/old:str/new:str`、缺 `valid_from_chapter/risk_class`)删;§03.1 的版本(`target_id:int`、缺 `risk_class`、缺 `entity`)删;§06 的版本(只有 `op/target_id/old/new/reason/evidence_refs`)删;§08.2.2 的版本(用 `entity_canonical_name/field_path`、`risk_class` 含 `knowledge_edge` 旧拼写)删。

**(a) 权威定义**

**`value_shift` 不是 beat_type**,而是**每个 beat 都有的属性**:`value_start`/`value_end`(价值极性)。任一 beat 的价值转向 = `value_start → value_end`。

Pydantic 侧:

```python
class BeatType(str, Enum):
    setup = "setup"
    turn = "turn"
    payoff_beat = "payoff_beat"
    tension_point = "tension_point"
    hook = "hook"

class Beat(BaseModel):
    beat_type: BeatType
    value_start: Optional[str] = None
    value_end: Optional[str] = None
    tension_level: Optional[int] = Field(default=None, ge=1, le=10)
```

**§02 必改项**:§02.6.1 `beat_type` CHECK 从 `('payoff_beat','hook','value_shift','tension_point','setup','turn')` 去掉 `value_shift`,定为五值;`value_from`/`value_to` 列改名 `value_start`/`value_end`。

**§05 必改项**:§05 中 `rising` / `tension` / `payoff` 等 beat 命名**重命名对齐**:`rising`/`tension` → `tension_point`,`payoff` → `payoff_beat`,价值转向用 beat 属性 `value_start/value_end` 表达而非独立 beat 类型。

**(b) 为何这样裁定**

§02.6.1 把 `value_shift` 列进 `beat_type` CHECK,同时又有 `value_from/value_to` 列——逻辑自指:既然每个 beat 都能有 value_from→value_to,那"value_shift 类型的 beat"和"带 value 变化的 setup beat"无法区分,且 §05 还另用 `rising/tension/payoff` 一套命名。McKee 的叙事理论里 value shift(价值转向)是**任何**节拍的可测属性(每个 beat 都在推动价值正负翻转),不是与 setup/turn 并列的一种节拍。故 `beat_type` 收敛为结构性的五类 `{setup, turn, payoff_beat, tension_point, hook}`,value 极性降为列 `value_start/value_end`(改名以避免与旧 `value_from/value_to` 的"故事内时间"语义混淆)。§05 的同义命名对齐到这五值。

#### R11 — `knowledge_edges` DDL 增两列 + `is_public` 定义

**(a) 权威定义**

```sql
-- §02.5.3 knowledge_edges 权威 DDL(增 public_from_chapter + secrecy_level)
CREATE TABLE IF NOT EXISTS knowledge_edges (
    id               TEXT PRIMARY KEY,
    knower_entity_id TEXT NOT NULL,                -- 知情者 → entities(id)
    secret_key       TEXT NOT NULL,                -- 秘密标识(如 'lin_real_identity')
    secret_fact_id   TEXT,                         -- → facts(id)(秘密内容本身,TEXT R2)
    knowledge_state  TEXT NOT NULL
        CHECK(knowledge_state IN ('knows','suspects','unaware','misinformed')),
    learned_chapter  INTEGER NOT NULL,             -- 故事内获知章
    public_from_chapter INTEGER,                   -- 【新增】该秘密对世界公开的起始章;NULL=尚属秘密
    secrecy_level    TEXT,                         -- 【新增】保密级别(见 CHECK)
    source           TEXT,                         -- 如何得知(对白/亲见/推理)
    fact_id          TEXT,                         -- TEXT R2
    created_at       TEXT NOT NULL DEFAULT (datetime('now')),
    FOREIGN KEY(knower_entity_id) REFERENCES entities(id),
    UNIQUE(knower_entity_id, secret_key, learned_chapter),
    CHECK(secrecy_level IS NULL OR secrecy_level IN ('public','open_secret','secret','top_secret'))

CREATE INDEX IF NOT EXISTS idx_know_knower  ON knowledge_edges(knower_entity_id, secret_key, learned_chapter);
CREATE INDEX IF NOT EXISTS idx_know_secret  ON knowledge_edges(secret_key);
CREATE INDEX IF NOT EXISTS idx_know_public  ON knowledge_edges(secret_key, public_from_chapter);
```

`is_public` 判定(供 §04 validator 调用,纯确定性):

```python
def is_public(secret_key: str, chapter: int, conn) -> bool:
    """某秘密在给定章是否已对世界公开。
    依据:存在该 secret_key 的边,其 public_from_chapter 非空且 <= chapter。"""
    row = conn.execute(
        "SELECT MIN(public_from_chapter) AS pf FROM knowledge_edges "
        "WHERE secret_key = ? AND public_from_chapter IS NOT NULL",
        (secret_key,),
    ).fetchone()
    return row["pf"] is not None and row["pf"] <= chapter
```

判定式(权威表述):
> `is_public(info, chapter)` ⟺ `public_from_chapter IS NOT NULL AND public_from_chapter <= chapter`

`secrecy_level` 取值:`{public, open_secret, secret, top_secret}`(NULL 视为未声明,按 `secret` 兜底)。

**§02 必改项**:§02.5.3 `knowledge_edges` 加 `public_from_chapter INTEGER`、`secrecy_level TEXT` 两列 + CHECK + 索引(如上)。
**§04 关联**:validator(知情者图/信息差)引用的 `public_from_chapter`、`secrecy_level`、`is_public()` 自此真实存在,可据此判"某秘密已公开则角色知情不算缺失边"。

**(b) 为何这样裁定**

§02.5.3 原 `knowledge_edges` 只有 `knowledge_state/learned_chapter`,能表达"谁在第几章知道",但**无法表达"这个秘密整体何时对世界公开"**。而 §04 的信息差 validator 需要判断:角色在第 N 章提及某信息,若该信息此时已**公开**,则不算"缺失边违规"(公开信息人人可知);只有当信息仍是秘密、而某未获知角色却表现得知情时,才报 hard issue。缺 `public_from_chapter` 这列,validator 引用的字段不存在(代码读到 NULL/报错)。补两列:`public_from_chapter`(秘密转公开的故事内时点,NULL=仍秘密)给出公开判定的确定性依据;`secrecy_level` 给出保密分级(影响泄密的风险评估与 require_human_for 命中)。`is_public` 用 `MIN(public_from_chapter)` 聚合是因为同一 secret_key 可能有多条边(多个知情者),只要存在一条公开边即视为已公开。
