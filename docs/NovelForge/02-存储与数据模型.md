## 2. 存储与数据模型(完整SQLite DDL)

> 本节给出 `novel.db` 的全量 SQLite DDL、关键索引、示例行,以及配套的 Pydantic 模型。控制平面/数据平面/治理平面的职责划分详见第01节;校验器(validator)如何消费这些表详见第04节;晋升闸门(PromotionPolicy)如何写入 staging/审计表详见第03节;Recall 的 as-of 投影如何查询这些表详见第06节。

### 2.1 存储设计总则(单一真相源)

**硬约束(对应11条硬原则之2/9/11):**

1. **单一真相源 = SQLite。** 业务表 + canon 账本 + World State Store + 工艺层 + 治理表 + FTS5(jieba)+ sqlite-vec(vec0)**全部同库同事务**,落在唯一一个 `novel.db` 文件(WAL 模式)。不存在第二个权威数据源。
2. **markdown 是只读渲染产物。** `story_bible.md`、各种卡片导出文件,一律是从 SQLite 表**确定性渲染**出来的只读视图(详见第03节渲染管线)。它们**不可手改、永不被 LLM 写回、可随时删除重渲**。任何"修改设定"的操作只能落到 `facts` / `fact_revisions`(只追加 + 状态变更),绝不 patch markdown。
3. **索引可重建。** `facts_fts`(FTS5)与 `scene_vec`(sqlite-vec)是**派生索引**,可丢可坏可一键重放:从 `facts` / L1 原子表 / L2 场景表重新生成。因此它们**不是真相源**,只是查询加速结构。
4. **草稿(L0)存文件,表里只存路径。** 大体量的原始草稿正文走文件系统(`drafts/` 目录),数据库 `draft_index` 表只存路径 + 元数据 + 校验和,避免 db 膨胀且便于 git 友好 diff。
5. **永不物理删除 canon。** `facts` 通过 `status`(canon / tentative / retconned)与 `fact_revisions`(append-only)表达生命周期;"删除"= 写一条 `deprecate`/`retconned` 修订,而非 `DELETE`。

**可重建性的版本绑定(对应硬原则11):**

派生索引依赖外部模型/分词器,**这些版本号必须随数据入库**,否则"重建"会得到与历史不一致的索引:

- `scene_vec` 的每条向量行,在其源表 `l2_scenes` 上记录 `embedding_model`(如 `bge-m3`)、`embedding_dim`、`embedding_version`(如 `2024-05`)。**重建时按 model+version 分组重算**,跨版本不可混用同一 `vec0` 表(维度/语义空间不同)。
- `facts_fts` 的 jieba 预分词结果依赖 `tokenizer_version`(jieba 版本 + 自定义词典哈希),记录在 `meta_kv` 表的全局元数据里。词典变更 ⇒ `tokenizer_version` 变更 ⇒ 触发全量重切词重建 `facts_fts`。
- 全局元数据表 `meta_kv` 统一保存 `schema_version`、`tokenizer_version`、`default_embedding_model`、`default_embedding_version`,作为"按版本重建"的单一查询入口。

```sql
-- 全局元数据 / 版本绑定(单行 KV,索引重建的依据)
CREATE TABLE IF NOT EXISTS meta_kv (
    key   TEXT PRIMARY KEY,
    value TEXT NOT NULL,
    updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);
-- 示例行:
-- ('schema_version', '2', ...)
-- ('tokenizer_version', 'jieba-0.42.1+dict_sha256:ab12...', ...)
-- ('default_embedding_model', 'bge-m3', ...)
-- ('default_embedding_version', '2024-05', ...)
```

**连接初始化 PRAGMA(每次开库执行):**

```sql
PRAGMA journal_mode = WAL;       -- 连写形态:读写不互斥
PRAGMA synchronous  = NORMAL;    -- WAL 下兼顾安全与吞吐
PRAGMA foreign_keys = ON;        -- 强制外键约束(默认关闭,必须显式开)
PRAGMA busy_timeout = 5000;      -- 后台 PipelineManager 与 API 并发时避免立即 SQLITE_BUSY
PRAGMA temp_store  = MEMORY;
-- 备份:周期性 .backup 到 novel.db.bak(WAL checkpoint 后);详见第08节运维
```

**通用列约定(所有表遵守):**

- 主键统一 `id TEXT PRIMARY KEY`,值为应用层生成的前缀化 ID(如 `fact_01H...`、`ent_01H...`),便于跨表引用与日志可读;ULID/UUIDv7 皆可。
- 时间戳分两类且**严格区分**:`created_at`/`updated_at` 是**真实世界写入时间**(`datetime('now')`,字符串 ISO8601);`*_chapter` / `story_time_*` 是**故事内时序**(整数章号 / 绝对 story_time),二者**不可混用**——一致性判断只认故事内时序。
- 枚举值用 `CHECK(col IN (...))` 落到 schema 层,validator 与 DDL 双重保障。
- 数组/嵌套结构(如 `aliases`、`evidence_refs`)统一存 `TEXT` 的 JSON,配 `json_valid()` CHECK;关系型查询仍走规范化表(如 `entity_aliases`),JSON 仅作冗余快照/渲染便利。

### 2.2 canon 真相源账本(facts / fact_revisions)

canon 是**只追加的结构化账本**(硬原则2)。`facts` 是每个事实的当前游标(指向最新生效修订 + 当前状态),`fact_revisions` 是该事实的完整 append-only 历史。LLM **绝不直接写这两张表**——它只产出 `BibleChangeProposal`(结构化 diff),经治理平面晋升后由系统写入(详见第03节)。

```sql
-- 2.2.1 facts:canon 事实的"当前视图"游标(每个逻辑事实一行)
CREATE TABLE IF NOT EXISTS facts (
    id              TEXT PRIMARY KEY,              -- fact_xxx,逻辑事实稳定 ID(跨修订不变)
    entity_id       TEXT,                          -- 关联实体(可空:全局世界规则等)
    fact_type       TEXT NOT NULL,                 -- 事实分类(见 CHECK)
    subject         TEXT NOT NULL,                 -- 主体(规范名或全局键)
    predicate       TEXT NOT NULL,                 -- 谓词/属性键,如 'power_rank'、'knows_secret'
    object          TEXT NOT NULL,                 -- 值(标量;复杂值放 detail_json)
    detail_json     TEXT,                          -- 结构化补充(JSON;CHECK json_valid)
    status          TEXT NOT NULL DEFAULT 'tentative'
                        CHECK(status IN ('canon','tentative','retconned')),
    valid_from_chapter INTEGER NOT NULL,           -- 故事内生效起始章(as-of 投影核心)
    valid_to_chapter   INTEGER,                    -- 失效章(NULL=至今有效);retcon 时回填
    current_revision_id TEXT NOT NULL,             -- 指向 fact_revisions 最新一条
    confidence      REAL,                          -- 仅排序用,不作晋升闸门(硬原则8)
    risk_tier       TEXT NOT NULL DEFAULT 'low'
                        CHECK(risk_tier IN ('low','medium','high')),  -- 晋升分级用
    created_at      TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at      TEXT NOT NULL DEFAULT (datetime('now')),
    FOREIGN KEY(entity_id) REFERENCES entities(id),
    CHECK(detail_json IS NULL OR json_valid(detail_json)),
    CHECK(fact_type IN (
        'world_rule','power_system','character_trait','relationship',
        'knowledge','event','item','location','numeric','style','misc'))
);

CREATE INDEX IF NOT EXISTS idx_facts_entity      ON facts(entity_id);
CREATE INDEX IF NOT EXISTS idx_facts_status      ON facts(status);
CREATE INDEX IF NOT EXISTS idx_facts_validrange  ON facts(valid_from_chapter, valid_to_chapter);
CREATE INDEX IF NOT EXISTS idx_facts_subj_pred   ON facts(subject, predicate);
CREATE INDEX IF NOT EXISTS idx_facts_type        ON facts(fact_type);

-- 示例行(facts):
-- id='fact_pw_lin', entity_id='ent_linfeng', fact_type='power_system',
--  subject='林枫', predicate='power_rank', object='筑基期', status='canon',
--  valid_from_chapter=12, valid_to_chapter=NULL, current_revision_id='rev_003',
--  risk_tier='high', confidence=0.95
```

```sql
-- 2.2.2 fact_revisions:append-only 完整修订历史(防漂移审计,硬原则9)
CREATE TABLE IF NOT EXISTS fact_revisions (
    id              TEXT PRIMARY KEY,              -- rev_xxx
    fact_id         TEXT NOT NULL,                 -- 所属逻辑事实
    revision_no     INTEGER NOT NULL,              -- 单调递增,从 1 起
    op              TEXT NOT NULL
                        CHECK(op IN ('add','update','deprecate','retcon','revert')),
    old_object      TEXT,                          -- 变更前值(add 时 NULL)
    new_object      TEXT,                          -- 变更后值(deprecate 时可 NULL)
    old_status      TEXT,
    new_status      TEXT NOT NULL,
    valid_from_chapter INTEGER NOT NULL,           -- 本修订声明的生效章
    reason          TEXT NOT NULL,                 -- 人/机给出的变更理由
    evidence_refs   TEXT,                          -- JSON 数组:出处引用(章节/原句锚点)
    actor           TEXT NOT NULL,                 -- 'human:<user>' / 'system:promotion'
    policy_mode     TEXT,                          -- 'human_gate'|'auto_promote'|'hybrid'
    source_candidate_id TEXT,                       -- 溯源到 fact_candidates(若来自 staging)
    created_at      TEXT NOT NULL DEFAULT (datetime('now')),
    FOREIGN KEY(fact_id) REFERENCES facts(id),
    UNIQUE(fact_id, revision_no),
    CHECK(evidence_refs IS NULL OR json_valid(evidence_refs))
);

CREATE INDEX IF NOT EXISTS idx_factrev_fact ON fact_revisions(fact_id, revision_no);
CREATE INDEX IF NOT EXISTS idx_factrev_op   ON fact_revisions(op);

-- 示例行(fact_revisions):
-- id='rev_003', fact_id='fact_pw_lin', revision_no=3, op='update',
--  old_object='炼气九层', new_object='筑基期', old_status='canon', new_status='canon',
--  valid_from_chapter=12, reason='第12章突破描写', evidence_refs='[{"chapter":12,"anchor":"§3.p2"}]',
--  actor='human:chayodom', policy_mode='human_gate', source_candidate_id='cand_88'
```

> **回滚 = 再 append(硬原则9):** `revert` 不删除任何历史,而是写一条新 `op='revert'` 的修订,把 `facts.current_revision_id`/`status`/`object` 指回目标旧值,并在 `promotion_log`(2.4)留痕。按 entity 的变更时间线 = `SELECT * FROM fact_revisions r JOIN facts f ON r.fact_id=f.id WHERE f.entity_id=? ORDER BY r.created_at`。

### 2.3 治理:staging(fact_candidates)

所有 LLM 产出先进 staging,**永不直达 canon**(硬原则5)。`fact_candidates` 是治理平面入口,状态机:`proposed → pending_review → (promoted | rejected | superseded)`。`promoted` = 已离开 staging 进入 canon(终态);HOLD 不迁移,停留 `proposed`。禁用 approved/committed/canon 作候选状态。

```sql
-- 2.3.1 fact_candidates:LLM 产出的待晋升候选(staging)
CREATE TABLE IF NOT EXISTS fact_candidates (
    candidate_id    TEXT PRIMARY KEY,              -- cand_xxx
    proposal_json   TEXT NOT NULL,                 -- BibleChangeProposal 完整序列化(见 2.9)
    op              TEXT NOT NULL
                        CHECK(op IN ('add','update','deprecate','retcon')),
    target_fact_id  TEXT,                          -- update/deprecate/retcon 指向的现有 fact(add 为 NULL)
    fact_type       TEXT NOT NULL,
    entity_id       TEXT,
    proposed_object TEXT,
    risk_tier       TEXT NOT NULL DEFAULT 'low'
                        CHECK(risk_tier IN ('low','medium','high')),
    confidence      REAL,                          -- 仅排序(硬原则8)
    evidence_strength REAL,                        -- 可程序校验的出处强度(晋升主依据)
    evidence_refs   TEXT,                          -- JSON 数组
    conflict_flags  TEXT,                          -- JSON:validator 检出的冲突摘要(空=无冲突)
    status          TEXT NOT NULL DEFAULT 'proposed'
                        CHECK(status IN ('proposed','pending_review','promoted','rejected','superseded')),
    source_chapter  INTEGER,                        -- 来自哪一章草稿的抽取
    source_skill    TEXT,                           -- 产出 Skill 名(可追溯)
    decided_by      TEXT,                            -- 'system:auto' / 'human:<user>'
    decided_at      TEXT,
    committed_revision_id TEXT,                      -- 晋升成功后回填 fact_revisions.id
    created_at      TEXT NOT NULL DEFAULT (datetime('now')),
    FOREIGN KEY(target_fact_id) REFERENCES facts(id),
    FOREIGN KEY(entity_id) REFERENCES entities(id),
    CHECK(proposal_json IS NULL OR json_valid(proposal_json)),
    CHECK(evidence_refs IS NULL OR json_valid(evidence_refs)),
    CHECK(conflict_flags IS NULL OR json_valid(conflict_flags))
);

CREATE INDEX IF NOT EXISTS idx_cand_status   ON fact_candidates(status);
CREATE INDEX IF NOT EXISTS idx_cand_risk     ON fact_candidates(risk_tier);
CREATE INDEX IF NOT EXISTS idx_cand_chapter  ON fact_candidates(source_chapter);
-- 晋升排序:无冲突优先 + evidence_strength 降序(confidence 仅次级排序)
CREATE INDEX IF NOT EXISTS idx_cand_rank     ON fact_candidates(status, evidence_strength DESC);

-- 示例行:
-- candidate_id='cand_88', op='update', target_fact_id='fact_pw_lin', fact_type='power_system',
--  entity_id='ent_linfeng', proposed_object='筑基期', risk_tier='high',
--  confidence=0.93, evidence_strength=0.9, conflict_flags='[]',
--  status='promoted', source_chapter=12, source_skill='ExtractSkill',
--  decided_by='human:chayodom', committed_revision_id='rev_003'
```

> 高风险(`risk_tier='high'`,即触及 world_rule/power_system/character_death/foreshadow_payoff/knowledge_edge_change)在 `auto_promote` 模式下仍**强制人审**:PromotionPolicy 读 `config.canon_governance.require_human_for[]`,命中则无论模式一律 `enqueue_review`(写 2.4 的 `review_queue`)。决策逻辑详见第03节。

### 2.4 治理:审计与审核队列(promotion_log / review_queue)

```sql
-- 2.4.1 promotion_log:晋升决策的 append-only 审计(只增不改,硬原则9)
-- 这是 promotion_log 的唯一权威 DDL(契约见第10节);§03 不再重复建表,只引用本表。
CREATE TABLE IF NOT EXISTS promotion_log (
    id              TEXT PRIMARY KEY,              -- plog_xxx
    candidate_id    TEXT NOT NULL,                 -- 关联 fact_candidates
    fact_id         TEXT,                          -- 命中的 canon 事实(若已落地)
    entity_id       TEXT,                          -- 关联实体(可空)
    decision        TEXT NOT NULL
                        CHECK(decision IN ('commit_canon','enqueue_review','hold_staging','reject','revert')),
    policy_mode     TEXT NOT NULL
                        CHECK(policy_mode IN ('human_gate','auto_promote','hybrid')),
    risk_tier       TEXT NOT NULL,
    evidence_strength REAL,
    chapter         INTEGER,                        -- 决策对应的故事内章
    conflict_summary TEXT,                          -- JSON
    old_value       TEXT,                          -- revert/update 留痕
    new_value       TEXT,
    reason          TEXT NOT NULL,
    actor           TEXT NOT NULL,                 -- 'system:policy' / 'human:<user>'
    reverts_log_id  TEXT,                          -- revert 时指向被回滚的 promotion_log 行
    created_at      TEXT NOT NULL DEFAULT (datetime('now')),
    FOREIGN KEY(candidate_id)   REFERENCES fact_candidates(candidate_id),
    FOREIGN KEY(reverts_log_id) REFERENCES promotion_log(id)
);
CREATE INDEX IF NOT EXISTS idx_plog_candidate ON promotion_log(candidate_id);
CREATE INDEX IF NOT EXISTS idx_plog_fact      ON promotion_log(fact_id);
CREATE INDEX IF NOT EXISTS idx_plog_time      ON promotion_log(created_at);

-- append-only 触发器:禁止 UPDATE/DELETE(防漂移审计,硬原则9)
CREATE TRIGGER IF NOT EXISTS trg_plog_no_update
    BEFORE UPDATE ON promotion_log
    BEGIN SELECT RAISE(ABORT, 'promotion_log is append-only'); END;
CREATE TRIGGER IF NOT EXISTS trg_plog_no_delete
    BEFORE DELETE ON promotion_log
    BEGIN SELECT RAISE(ABORT, 'promotion_log is append-only'); END;
-- 注:不存在 canon_changelog 表。canon 内容变更流(改了什么值/状态)的职责由 fact_revisions(2.2.2)承担;
--     promotion_log 只记录"晋升决策"这一治理事件。两者职责分离,不重复建账本。

-- 2.4.2 review_queue:human_gate 待人审任务队列
CREATE TABLE IF NOT EXISTS review_queue (
    id              TEXT PRIMARY KEY,              -- rq_xxx
    candidate_id    TEXT NOT NULL,
    priority        INTEGER NOT NULL DEFAULT 100,  -- 越小越急(high risk → 低值)
    risk_tier       TEXT NOT NULL,
    reason          TEXT NOT NULL,                 -- 入队原因(命中 require_human_for 等)
    status          TEXT NOT NULL DEFAULT 'pending'
                        CHECK(status IN ('pending','in_review','approved','rejected','expired')),
    assigned_to     TEXT,
    enqueued_at     TEXT NOT NULL DEFAULT (datetime('now')),
    resolved_at     TEXT,
    FOREIGN KEY(candidate_id) REFERENCES fact_candidates(candidate_id)
);
CREATE INDEX IF NOT EXISTS idx_rq_status   ON review_queue(status, priority);
CREATE INDEX IF NOT EXISTS idx_rq_candidate ON review_queue(candidate_id);

-- 示例行(review_queue):
-- id='rq_12', candidate_id='cand_88', priority=10, risk_tier='high',
--  reason='require_human_for: power_system', status='approved'
```

### 2.5 World State Store(确定性硬状态层)

World State Store 支撑 `get_world_state(as_of_chapter=N)` 投影(硬原则3)与全部确定性 validator(硬原则1/7)。所有表都带**故事内时序**列以支持 as-of 查询;**永不删除,只追加状态变更**。

#### 2.5.1 entities(实体 + aliases)

```sql
CREATE TABLE IF NOT EXISTS entities (
    id              TEXT PRIMARY KEY,              -- ent_xxx
    canonical_name  TEXT NOT NULL UNIQUE,          -- 规范名(名字规范化的目标)
    entity_type     TEXT NOT NULL
                        CHECK(entity_type IN ('character','location','item','faction','concept')),
    first_appear_chapter INTEGER,
    status          TEXT NOT NULL DEFAULT 'active'
                        CHECK(status IN ('active','dead','retired','sealed')),
    detail_json     TEXT,
    created_at      TEXT NOT NULL DEFAULT (datetime('now')),
    CHECK(detail_json IS NULL OR json_valid(detail_json))
);
CREATE INDEX IF NOT EXISTS idx_entities_type ON entities(entity_type);

-- 别名表:名字规范化(MVP 硬状态最小集之一,硬原则7)。规范化 = alias → canonical_name
CREATE TABLE IF NOT EXISTS entity_aliases (
    id              TEXT PRIMARY KEY,
    entity_id       TEXT NOT NULL,
    alias           TEXT NOT NULL,                 -- 别名/绰号/旧名/称谓
    alias_type      TEXT NOT NULL DEFAULT 'nickname'
                        CHECK(alias_type IN ('nickname','formal','title','old_name','typo')),
    valid_from_chapter INTEGER,                     -- 该称谓何时开始用(可空)
    FOREIGN KEY(entity_id) REFERENCES entities(id),
    UNIQUE(alias)                                   -- 全局唯一,保证规范化无歧义
);
CREATE INDEX IF NOT EXISTS idx_alias_entity ON entity_aliases(entity_id);

-- 示例:entities(id='ent_linfeng', canonical_name='林枫', entity_type='character')
--       entity_aliases(entity_id='ent_linfeng', alias='枫少', alias_type='nickname')
```

#### 2.5.2 power_ranks(有序枚举)+ character_power_log

境界单调性校验依赖**有序枚举**:`rank_order` 整数提供全序比较(硬原则1 的"全序比较"靠它,而非字符串)。

```sql
-- 境界体系定义:有序枚举(rank_order 越大越高)
CREATE TABLE IF NOT EXISTS power_ranks (
    id              TEXT PRIMARY KEY,
    system_name     TEXT NOT NULL,                 -- 体系名(可多体系并存)
    rank_name       TEXT NOT NULL,                 -- 境界名,如 '筑基期'
    rank_order      INTEGER NOT NULL,              -- 全序键:炼气=10,筑基=20,金丹=30...
    detail_json     TEXT,
    UNIQUE(system_name, rank_name),
    UNIQUE(system_name, rank_order)                -- 同体系内 order 不重复
);
CREATE INDEX IF NOT EXISTS idx_prank_order ON power_ranks(system_name, rank_order);

-- 角色境界变更日志:append-only,as-of 投影取 <= N 的最新一条
CREATE TABLE IF NOT EXISTS character_power_log (
    id              TEXT PRIMARY KEY,
    entity_id       TEXT NOT NULL,
    system_name     TEXT NOT NULL,
    rank_id         TEXT NOT NULL,                 -- → power_ranks
    rank_order      INTEGER NOT NULL,              -- 冗余:免 join 即可做单调性校验
    change_chapter  INTEGER NOT NULL,              -- 故事内生效章
    change_type     TEXT NOT NULL
                        CHECK(change_type IN ('breakthrough','injury_drop','seal','unseal','init')),
    fact_id         TEXT,                          -- 溯源 canon
    created_at      TEXT NOT NULL DEFAULT (datetime('now')),
    FOREIGN KEY(entity_id) REFERENCES entities(id),
    FOREIGN KEY(rank_id)   REFERENCES power_ranks(id)
);
CREATE INDEX IF NOT EXISTS idx_cpl_entity ON character_power_log(entity_id, change_chapter);

-- 境界单调性校验(纯 SQL,无 LLM):同一 entity 后一条 rank_order 不得 < 前一条(除 injury_drop/seal)
-- 示例:entity 在 ch12 breakthrough 到 rank_order=20;若 ch15 出现 rank_order=10 且非 injury_drop ⇒ hard issue
```

#### 2.5.3 knowledge_edges(知情者图)

信息差/知情者图(硬原则1 的"信息差"、硬原则5 的"知情者图变更需人审"):谁在第几章知道了什么秘密。validator 检"缺失边"(角色未知却表现得知情)。

```sql
CREATE TABLE IF NOT EXISTS knowledge_edges (
    id              TEXT PRIMARY KEY,
    knower_entity_id TEXT NOT NULL,                -- 知情者
    secret_key      TEXT NOT NULL,                 -- 秘密标识(如 'lin_real_identity')
    secret_fact_id  TEXT,                          -- → facts(秘密内容本身)
    knowledge_state TEXT NOT NULL
                        CHECK(knowledge_state IN ('knows','suspects','unaware','misinformed')),
    learned_chapter INTEGER NOT NULL,              -- 故事内获知章
    source          TEXT,                          -- 如何得知(对白/亲见/推理)
    public_from_chapter INTEGER,                    -- 该秘密何章起公开(NULL=尚属秘密)
    secrecy_level   TEXT
                        CHECK(secrecy_level IS NULL OR secrecy_level IN ('public','open_secret','secret','top_secret')),
    fact_id         TEXT,
    created_at      TEXT NOT NULL DEFAULT (datetime('now')),
    FOREIGN KEY(knower_entity_id) REFERENCES entities(id),
    UNIQUE(knower_entity_id, secret_key, learned_chapter)
);
CREATE INDEX IF NOT EXISTS idx_know_knower ON knowledge_edges(knower_entity_id, secret_key, learned_chapter);

### 2.8 检索索引:FTS5(facts_fts + drafts_fts + jieba 预分词)与 sqlite-vec(scene_vec,仅 L2)

检索以实体/关系结构化 SQL 为主、语义为辅(硬原则4);BM25 作关键词补充;向量仅作"查相似桥段"的可选增强且仅对 L2 建索引(硬原则4/11)。全文检索拆两张表:`facts_fts` 只索引 **canon facts** 的可检索文本列;`drafts_fts` 索引 **L0 草稿正文行**(用于"查相似桥段/原文检索")。**三者(含可选 `facts_fts_tri`)皆为可重建派生索引,非真相源。**

#### 2.8.1 facts_fts(只索引 canon facts;FTS5 + jieba 预分词)

`facts_fts` **只索引 `status='canon'` 的事实**(tentative/retconned 不入索引,由重建逻辑过滤;晋升/失效时增删对应行)。SQLite 自带分词器对中文按字切分、召回噪声大。**工程方案:在应用层用 jieba 预分词**,把 `subject/predicate/object/detail` 切成空格分隔的词串写入 FTS5,FTS5 侧用最简单的 `unicode61` 兼容空格切分。这样 BM25 在词粒度上计算,召回质量显著优于裸字符。

**关键工程点(jieba 预分词):**

1. **预分词列分离。** `facts_fts` 不直接挂 `content='facts'` 取原始列(否则 FTS5 看到的是未切词中文)。改为存储**预切词后的文本**:应用层 `jieba.lcut()` 后用空格 join,再写入 FTS5。查询端**同样对查询词 jieba 切词**后再构造 MATCH,保证 query/doc 同口径。
2. **tokenizer 用 `unicode61 remove_diacritics 2`**,因为输入已是空格分隔的词,FTS5 只需按空白切分。
3. **版本绑定。** jieba 版本 + 自定义词典(人名/境界/地名等专名词典,从 `entities`/`entity_aliases`/`power_ranks` 导出)的哈希 = `tokenizer_version`,存 `meta_kv`。词典变更 ⇒ 全量重切词重建 `facts_fts`。专名入词典对网文专有名词召回至关重要。
4. **重建。** `facts_fts` 可随时 `DROP + 重建`:遍历 `facts` 中 `status='canon'` 的行,逐行 jieba 切词后重灌。因此它**不是真相源**。

CREATE VIRTUAL TABLE IF NOT EXISTS facts_fts USING fts5(

    object_tok,                 -- jieba 预分词:值
    detail_tok,                 -- jieba 预分词:detail_json 抽取的可读文本
    tokenize = "unicode61 remove_diacritics 2"

```

#### 2.8.2 drafts_fts(索引 L0 草稿正文行;FTS5 + jieba 预分词)

`drafts_fts` 用于"查相似桥段/原文检索",索引 L0 草稿的正文行(行区间映射回 `draft_index`/`drafts/` 文件)。与 `facts_fts` 同口径:应用层 jieba 预分词(写入前空格 join,查询同口径),`tokenizer_version` 入 `meta_kv`。

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS drafts_fts USING fts5(
    draft_id    UNINDEXED,      -- 回连 draft_index.id(不参与全文索引)
    chapter_no  UNINDEXED,      -- 章号(过滤用)
    line_start  UNINDEXED,      -- 正文行区间起
    line_end    UNINDEXED,      -- 正文行区间止
    body,                       -- jieba 预分词后的正文行文本
    tokenize = "unicode61 remove_diacritics 2"
);
-- 查询("查相似桥段",查询词同样 jieba 切词):
-- SELECT draft_id, chapter_no, line_start, line_end, bm25(drafts_fts) AS score
-- FROM drafts_fts WHERE drafts_fts MATCH :jieba_query
--   AND chapter_no <= :as_of_chapter            -- 配合 as-of:不召回未来章节
-- ORDER BY score LIMIT 20;
```

> 否定型/全局禁忌(如"20章前不暴露反派身份")**不走检索**,直接 always-on 硬注入 system prompt(硬原则4),不入 `facts_fts`。RRF(k=60,客户端,只用排名不归一化)融合 BM25 与向量排名的逻辑在 Recall 层实现,详见第06节。

#### 2.8.3 scene_vec(sqlite-vec / vec0,仅 L2 场景块)

向量索引**只对 L2 场景块建**(硬原则11),用于"查相似桥段"。源表 = `l2_scenes`;`scene_vec` 可丢可从 `l2_scenes` 按 `embedding_model`+`embedding_version` 重算重建。维度由模型决定(示例 bge-m3 = 1024),**跨模型/版本不可混用同一 vec0 表**。

```sql
-- sqlite-vec 虚拟表(vec0):仅 L2 场景向量
-- scene_rowid 与 l2_scenes 的整数映射键对齐(应用层维护映射,或用 l2_scenes 的整型代理键)
CREATE VIRTUAL TABLE IF NOT EXISTS scene_vec USING vec0(
    scene_rowid     INTEGER PRIMARY KEY,          -- 对应 l2_scenes 行
    chapter         INTEGER partition key,         -- 分区键:常按章节范围过滤
    scene_embedding float[1024],                   -- 维度随 embedding_model 固定(bge-m3=1024)
    embedding_version TEXT,                         -- 元数据列:支持按版本过滤/重建
    +scene_id       TEXT                            -- 辅助列(+):回连 l2_scenes.id,免二次 join
);

-- KNN 查询("查相似桥段",可选增强,推后):
-- SELECT scene_id, distance
-- FROM scene_vec
-- WHERE scene_embedding MATCH :query_vec
--   AND k = 10
--   AND chapter <= :as_of_chapter            -- 配合 as-of:不召回未来章节
--   AND embedding_version = :current_version; -- 只查当前版本空间
```

> **重建流程**(索引可丢可重放,硬原则11):`DROP TABLE scene_vec` → 按 `meta_kv.default_embedding_model/version` 选取 `l2_scenes` 行 → 对每行重算 embedding(同 model+version)→ 重灌 `scene_vec`,并回填 `l2_scenes.indexed_at`。因维度/语义空间由 model+version 决定,**升级模型 = 新建一张 vec0 表全量重灌**,不可与旧向量混存。

### 2.9 关键 Pydantic 模型(Fact / BibleChangeProposal / FactCandidate)

结构化抽取与提案统一用 instructor + Pydantic,`field_validator` 做自愈重试(硬原则11)。下列模型是 LLM 输出契约与 DB 行的桥梁:LLM 只产出 `BibleChangeProposal`(绝不写 bible),经治理落 `fact_candidates`,晋升后落 `facts`/`fact_revisions`。

```python
from __future__ import annotations
from enum import Enum
from typing import Optional, Literal
from pydantic import BaseModel, Field, field_validator, model_validator


# ---- 共享枚举(与 DDL 的 CHECK 严格对齐) ----
class FactStatus(str, Enum):
    canon = "canon"
    tentative = "tentative"
    retconned = "retconned"

class FactType(str, Enum):
    world_rule = "world_rule"; power_system = "power_system"
    character_trait = "character_trait"; relationship = "relationship"
    knowledge = "knowledge"; event = "event"; item = "item"
    location = "location"; numeric = "numeric"; style = "style"; misc = "misc"

class RiskTier(str, Enum):
    low = "low"; medium = "medium"; high = "high"


class EvidenceRef(BaseModel):
    chapter: int                       # 故事内章号
    anchor: Optional[str] = None       # 原文锚点(段/句)
    quote: Optional[str] = None        # 可选原句,供人审核对


# ---- Fact:canon 真相源行(facts 表的应用层投影) ----
class Fact(BaseModel):
    id: str
    entity_id: Optional[str] = None
    fact_type: FactType
    subject: str
    predicate: str
    object: str
    detail: Optional[dict] = None              # 对应 detail_json
    status: FactStatus = FactStatus.tentative
    valid_from_chapter: int = Field(ge=0)      # 故事内时序,非真实时间
    valid_to_chapter: Optional[int] = None
    current_revision_id: str
    confidence: Optional[float] = Field(default=None, ge=0, le=1)  # 仅排序,不作闸门
    risk_tier: RiskTier = RiskTier.low

    @model_validator(mode="after")
    def _check_valid_range(self):
        if self.valid_to_chapter is not None and self.valid_to_chapter < self.valid_from_chapter:
            raise ValueError("valid_to_chapter 不得早于 valid_from_chapter")
        return self

    @model_validator(mode="after")
    def _high_risk_types(self):
        # 触及硬一致性的类型自动判高风险(晋升按风险分级,硬原则8)
        if self.fact_type in {FactType.world_rule, FactType.power_system} and self.risk_tier == RiskTier.low:
            self.risk_tier = RiskTier.high
        return self


# ---- BibleChangeProposal:LLM 唯一被允许产出的结构化 diff(绝不写 bible) ----
# 单一权威契约定义在第10节(数据契约与命名权威):字段 op/target_id/entity/fact_type/
# old/new/reason/evidence_refs/valid_from_chapter/risk_class,及 op↔target_id、evidence_refs
# 非空等校验,均以第10节为准。本节不再重复定义,直接 import 引用。
from novelforge.contracts import BibleChangeProposal  # 见第10节权威定义


# ---- FactCandidate:staging 行(fact_candidates 表的应用层投影) ----
class FactCandidate(BaseModel):
    candidate_id: str                          # 对应 fact_candidates.candidate_id(cand_xxx)
    proposal: BibleChangeProposal              # 对应 proposal_json
    op: Literal["add", "update", "deprecate", "retcon"]
    target_fact_id: Optional[str] = None
    fact_type: FactType
    entity_id: Optional[str] = None
    proposed_object: Optional[str] = None
    risk_tier: RiskTier = RiskTier.low
    confidence: Optional[float] = Field(default=None, ge=0, le=1)   # 仅排序
    evidence_strength: float = Field(default=0.0, ge=0, le=1)       # 晋升主依据
    conflict_flags: list[dict] = Field(default_factory=list)       # validator 检出的冲突(空=无冲突)
    status: Literal["proposed", "pending_review", "promoted", "rejected", "superseded"] = "proposed"
    source_chapter: Optional[int] = None
    source_skill: Optional[str] = None

    @property
    def is_promotable_to_canon(self) -> bool:
        # 晋升依据(硬原则8):无冲突 + 非高风险 + 出处够强;confidence 不参与
        return (
            not self.conflict_flags
            and self.risk_tier != RiskTier.high
            and self.evidence_strength >= 0.6
        )
```

> PromotionPolicy 如何消费 `FactCandidate.is_promotable_to_canon`、`risk_tier` 与 `config.canon_governance.require_human_for[]` 做 `commit_canon` vs `enqueue_review` 分叉,详见第03节;`evidence_strength` 的具体打分函数(出处可验权重最高)亦在第03节定义。

### 2.10 建表顺序与外键说明

由于开启了 `PRAGMA foreign_keys = ON`,建表须按依赖顺序(被引用表先建):

1. `meta_kv`
2. `entities` → `entity_aliases`
3. `power_ranks` → `character_power_log`
4. `geo_locations` → `travel_edges`、`timeline_events`
5. `facts`(引用 entities)→ `fact_revisions`
6. `knowledge_edges`、`item_ownership`、`item_log`、`gimmick_rules` → `gimmick_usage_log`、`numeric_facts`
7. `foreshadow` → `beats`、`chapter_cards`、`character_cards`、`pacing_state`
8. `draft_index` → `l1_atoms`、`l2_scenes`
9. `fact_candidates`(引用 facts/entities)→ `promotion_log`、`review_queue`
10. 派生索引最后建:`facts_fts`、`drafts_fts`、可选 `facts_fts_tri`、`scene_vec`(可随时 DROP 重建)

> `entities`↔`facts`、`facts`↔`fact_candidates` 存在前向引用(facts 引 entities,candidates 引 facts);若工具链严格校验,可对个别 FK 采用应用层软约束或延迟建索引。所有 `*_log` / `fact_revisions` / `promotion_log` 为 append-only,**任何业务逻辑不得发出 UPDATE/DELETE**(由仓储层封装强制),这是防漂移审计的基石(硬原则9)。
