### 11.1 `PromotionPolicy.detect_conflict(candidate) -> ConflictSet`

> 第 3 节 `PromotionPolicy.decide()` 调用此函数;返回非空 `ConflictSet` 即触发 `Route.REVIEW`(R5,"与既有 canon 冲突")。本函数**纯确定性、无 LLM**(硬原则 7)。

**用途**:对一个待晋升候选 `candidate`,按 `entity_id + fact_type + predicate` 锁定现有 canon facts,检出三类冲突:(a) 同实体同谓词不同值;(b) `valid_from/valid_to` 区间重叠冲突;(c) 与 World State 确定性 validator 冲突。

**输入**:
- `candidate`:`fact_candidates` 行(R3,`candidate_id` 为 `cand_xxx`)。关键字段 `entity_id`、`fact_type`、`proposed_object`、`proposal_json`(反序列化为 `BibleChangeProposal`,R8,含 `entity`/`fact_type`/`new`/`valid_from_chapter`/`evidence_refs`);`predicate` 从 `proposal_json.new.predicate` 取(与 `facts.predicate` 对齐,第 2 节 2.2.1)。
- `conn`:同库同事务 SQLite 连接。

**输出**:`ConflictSet`(Pydantic),`bool(conflict_set)` 为真即有冲突:

```python
class ConflictItem(BaseModel):
    kind: Literal["value_mismatch", "interval_overlap", "world_state"]  # (a)/(b)/(c)
    fact_id: Optional[str]                  # 命中的现有 canon fact(fact_xxx),(c) 类可为 None
    detail: str                             # 人类可读冲突摘要
    severity: Literal["hard", "soft"]       # 写入 promotion_log.conflict_summary

class ConflictSet(BaseModel):
    items: list[ConflictItem] = []
    def __bool__(self) -> bool: return len(self.items) > 0
    def has_hard(self) -> bool: return any(i.severity == "hard" for i in self.items)
```

**(a) 同实体同谓词不同值 — SQL**:在当前生效的 canon facts 中,同 `entity_id`+`fact_type`+`predicate` 但 `object` 不同。

```sql
-- :entity_id / :fact_type / :predicate / :proposed_object 来自 candidate
SELECT id AS fact_id, object AS existing_object, valid_from_chapter, valid_to_chapter
FROM facts
WHERE status = 'canon'                       -- R4:只与 canon 比对(tentative/retconned 不算冲突源)
  AND entity_id IS :entity_id
  AND fact_type = :fact_type
  AND predicate = :predicate
  AND object <> :proposed_object;
```
命中即产 `value_mismatch`。`severity` 由 §11.3 `classify_risk(candidate)` 决定:`high → hard`,否则 `soft`。

**(b) valid 区间重叠冲突 — SQL**:同实体同谓词的两条事实,若声明的 `[valid_from_chapter, valid_to_chapter]`(NULL=至今)区间**重叠**且值不同,则同一时点该实体该属性有两个值,矛盾。候选区间为 `[:cand_from, :cand_to]`(`update` 时 `cand_to` 取目标 fact 现值或 NULL)。

```sql
SELECT id AS fact_id, object AS existing_object, valid_from_chapter AS ef, valid_to_chapter AS et
FROM facts
WHERE status = 'canon'
  AND entity_id IS :entity_id
  AND fact_type = :fact_type
  AND predicate = :predicate
  AND id <> COALESCE(:target_fact_id, '')          -- update 自身不算冲突
  AND object <> :proposed_object
  -- 半开区间重叠:max(starts) < min(ends);NULL 视为 +∞
  AND :cand_from < COALESCE(et, 2147483647)
  AND ef         < COALESCE(:cand_to, 2147483647);
```
命中即产 `interval_overlap`。注:正常的"接续"(旧值在第 12 章被新值取代,旧值 `valid_to=11`、新值 `valid_from=12`)**不重叠**,不报;只有真重叠(同时段两值)才报。`update` 的合法路径应是先 `deprecate` 旧值(回填 `valid_to`)再 `add` 新值,由晋升事务保证(§11.7)。

**(c) 与 World State 确定性 validator 冲突**:候选若触及硬状态(境界/知情者图/物品/数值/时间线),把候选转成一条 claim,跑对应 validator(第 4 节 4.3),命中即冲突。**复用第 4 节既有 validator,不另写规则**:

```python
WORLD_STATE_TYPES = {"power_system", "knowledge", "item", "numeric", "event"}

def _world_state_conflict(cand, conn) -> list[ConflictItem]:
    if cand.fact_type not in WORLD_STATE_TYPES:
        return []
    world = get_world_state(as_of_chapter=cand.valid_from_chapter - 1, conn=conn)  # 第 4 节 4.4
    claim = candidate_to_claim(cand)            # 候选 → 一条 ContinuityClaim(第 4 节 4.2)
    issues = run_all_validators([claim], world, conn)          # 第 4 节 4.3.9
    return [ConflictItem(kind="world_state", fact_id=None,
                         detail=f"{i.code}: {i.message}",
                         severity="hard" if i.kind == "hard" else "soft")
            for i in issues if i.severity in ("major", "critical")]
```
典型命中:`POWER_LEVEL_SKIP`(境界越级,第 4 节 4.3.1)、`KNOWLEDGE_LEAK`(知情者越权,第 4 节 4.3.2,依赖 R11 的 `is_public`)。

**主体伪代码 + 默认参数**:

```python
class DetectConflictConfig(BaseModel):
    compare_status: tuple = ("canon",)        # 默认只与 canon 比对(R4)
    interval_inf: int = 2_147_483_647         # NULL valid_to 的 +∞ 哨兵
    world_state_min_severity: str = "major"   # (c) 类只采纳 major/critical

def detect_conflict(cand, conn, cfg=DetectConflictConfig()) -> ConflictSet:
    risk = classify_risk(cand)                                   # §11.3
    sev  = "hard" if risk == "high" else "soft"
    items = []
    for r in q_value_mismatch(conn, cand):                       # (a)
        items.append(ConflictItem(kind="value_mismatch", fact_id=r["fact_id"],
            detail=f"{cand.predicate}: 现={r['existing_object']} vs 候选={cand.proposed_object}",
            severity=sev))
    for r in q_interval_overlap(conn, cand, cfg):                # (b)
        items.append(ConflictItem(kind="interval_overlap", fact_id=r["fact_id"],
            detail=f"区间重叠 [{r['ef']},{r['et']}] ∩ [{cand.valid_from},{cand.valid_to}]",
            severity=sev))
    items += _world_state_conflict(cand, conn)                   # (c)
    return ConflictSet(items=items)
```

**小例子**:候选 `cand_88`(`op='update'`, `target_fact_id='fact_pw_lin'`, `entity_id='ent_linfeng'`, `fact_type='power_system'`, `predicate='power_rank'`, `proposed_object='元婴期'`, `valid_from_chapter=41`)。现有 canon `fact_pw_lin` = `筑基期`@valid_from=12。
- (a) 同实体同谓词值不同(`筑基期`≠`元婴期`)→ `value_mismatch`。但因为这是合法的"接续突破"(应走 deprecate+add),(b) 检查发现旧值 `valid_to` 将被回填为 40、新值 `valid_from`=41,**不重叠**→ (b) 不报。
- (c) `as-of(40)` 林枫=筑基(order=20),候选迁移到元婴(order=40),差 20 > `max_jump_per_step*rank_step`=10 → `POWER_LEVEL_SKIP`(major)→ `world_state` 冲突(hard)。
- 结果:`ConflictSet.has_hard() == True` → `PromotionPolicy` 路由 `REVIEW`。

### 11.2 `score_evidence(candidate) -> float`(值域 [0, 1])

> 第 3 节 `PromotionPolicy` 用此分数(写入 `fact_candidates.evidence_strength`)与 `config.canon_governance.auto_promote_threshold`(默认 0.82)比较,决定低风险候选能否 `COMMIT`。**强调:`confidence`(LLM 自报)绝不参与本分数,只作次级排序(硬原则 8,R 裁定一致)。**

**用途**:给候选打一个**可程序复算**的证据强度分,衡量"这条断言有多可信地源自正文"。

**公式**:

```
score = w_src * src_verifiable        # 出处可验:evidence_refs 指向的 L0 行区间真实存在 + 关键 token 命中
      + w_rec * recurrence_norm       # 复现度:跨章重复出现次数归一
      + w_cons * cross_consistency    # 交叉一致:与现有 canon 不冲突=1,有冲突=0
```

**默认权重**(与第 8 节 `config.canon_governance.promotion_ranking.weights` 对齐;此处采纳本节建议值 0.6/0.2/0.2,第 8 节示例 0.6/0.3/0.1 为旧值,以本节为准):

```python
class ScoreEvidenceConfig(BaseModel):
    w_src: float = 0.60          # 出处可验,权重最高
    w_rec: float = 0.20          # 跨章复现
    w_cons: float = 0.20         # 与 canon 交叉一致
    recurrence_cap: int = 3      # 复现归一上限:出现 >=3 次即满分
    key_token_min_hit: int = 1   # 行区间内至少命中几个关键 token 才算"出处可验"
    # confidence(LLM 自报)不在此处,硬原则 8
```

**三个子项的计算**:

- `src_verifiable ∈ {0, 1}`(MVP 取二值,可平滑为 [0,1]):遍历 `candidate.evidence_refs`(每项 `{draft_id, chapter, line_start, line_end}`,与 R6 `drafts_fts` 的行区间口径一致),(1) 该行区间在 `drafts`/`drafts_fts` 中真实存在;(2) 取 `proposed_object` 与实体名的关键 token,在该行区间正文中命中数 `>= key_token_min_hit`。任一 ref 满足即 1。

```sql
-- 验证一条 evidence_ref 的行区间存在,并取回正文供 token 命中检查(R6:drafts_fts 索引 L0 草稿正文行)
SELECT body FROM drafts_fts
WHERE draft_id = :draft_id AND chapter_no = :chapter
  AND line_start <= :ref_line_start AND line_end >= :ref_line_end;
```

- `recurrence_norm ∈ [0,1]`:同 `entity_id`+`predicate`(或同 `secret_key` 等键)的断言在多少**不同章**出现过(从 `fact_candidates` 历史 + `drafts_fts` BM25 命中计数),`min(count, recurrence_cap) / recurrence_cap`。

```sql
SELECT COUNT(DISTINCT source_chapter) AS n
FROM fact_candidates
WHERE entity_id IS :entity_id AND fact_type = :fact_type
  AND json_extract(proposal_json, '$.new.predicate') = :predicate;
```

- `cross_consistency ∈ {0, 1}`:`detect_conflict(candidate)`(§11.1)有 hard 冲突 → 0,否则 1。(soft 冲突不归零,只在路由层处理。)

**主体伪代码**:

```python
def score_evidence(cand, conn, cfg=ScoreEvidenceConfig()) -> float:
    src  = 1.0 if any(_ref_verifiable(r, cand, conn, cfg) for r in cand.evidence_refs) else 0.0
    rec  = min(_recurrence_count(cand, conn), cfg.recurrence_cap) / cfg.recurrence_cap
    cons = 0.0 if detect_conflict(cand, conn).has_hard() else 1.0
    score = cfg.w_src * src + cfg.w_rec * rec + cfg.w_cons * cons
    return round(min(max(score, 0.0), 1.0), 4)     # 钳到 [0,1]
```

**小例子**:候选"林枫=筑基期@ch12",`evidence_refs=[{draft_id:'d12', chapter:12, line_start:40, line_end:44}]`,该行区间正文含"林枫…踏入筑基" → token 命中 → `src=1.0`。该断言在 ch12、ch13、ch15 三章复现 → `count=3`,`rec = min(3,3)/3 = 1.0`。与 canon 无 hard 冲突 → `cons=1.0`。
`score = 0.6*1.0 + 0.2*1.0 + 0.2*1.0 = 1.0`。
反例:LLM 自报 `confidence=0.99` 但 `evidence_refs` 指向不存在的行区间 → `src=0.0`;仅出现 1 次 → `rec=1/3≈0.33`;无冲突 → `cons=1.0`。`score = 0.6*0 + 0.2*0.33 + 0.2*1.0 = 0.267`,远低于 0.82 阈值 → 不 auto,即便 confidence 极高(印证硬原则 8)。

### 11.3 `classify_risk(candidate) -> {"low", "medium", "high"}`

> 第 3 节 `PromotionPolicy` 与 §11.1/§11.2 复用;输出写入 `fact_candidates.risk_tier`(第 2 节 2.3.1)与 `BibleChangeProposal.risk_class`(R8)。命中 `high` 在 `auto_promote`/`hybrid` 模式下仍强制 `REVIEW`(R5)。**纯查表,无 LLM。**

**用途**:把候选按"改错的代价"分三级,驱动晋升路由与冲突 severity。

**输入**:`candidate`(其 `fact_type`、`entity_id`、`proposal_json`)、`config.canon_governance.require_human_for`(第 8 节,R10 拼写为 `knowledge_edge_change`)。

**输出**:`Literal["low","medium","high"]`。

**规则表**(自上而下,首个命中即返回):

| 优先级 | 条件 | 结果 | 依据 |
|---|---|---|---|
| 1 | `fact_type` 触及 World State **硬状态**:`world_rule` / `power_system` / `numeric`(关键数值)/ `item`(关键道具) | `high` | 硬原则 1/3;改错会让 as-of 投影与 validator 全线漂移 |
| 2 | 命中 `config.canon_governance.require_human_for`(默认含 `world_rule`/`power_system`/`character_death`/`foreshadow_payoff`/`knowledge_edge_change`,R10) | `high` | R5/R10;治理强制人审项 |
| 3 | `fact_type == 'relationship'` 或涉及**次要人物**事实(`entity` 非主角集)或 `event`(非死亡/非伏笔回收) | `medium` | 关系/次要人物错了可控、影响局部 |
| 4 | `fact_type ∈ {'style', 'misc'}` 或场景摘要类软记忆 | `low` | 文风/场景摘要,改错代价低 |
| 5 | 其余 | `medium` | 安全默认:拿不准就抬到 medium |

```python
HIGH_HARD_TYPES = {"world_rule", "power_system", "numeric", "item"}
LOW_TYPES       = {"style", "misc"}
MEDIUM_TYPES    = {"relationship", "event"}

def classify_risk(cand, cfg_governance) -> Literal["low", "medium", "high"]:
    require_human = set(cfg_governance.require_human_for)   # 含 'knowledge_edge_change'(R10)
    ft = cand.fact_type
    # 规则2 的统一映射:fact_type → require_human_for 键
    rh_key = {"power_system": "power_system", "world_rule": "world_rule",
              "knowledge": "knowledge_edge_change", "event": _event_rh_key(cand)}.get(ft)
    if ft in HIGH_HARD_TYPES:                  return "high"   # 规则1
    if rh_key and rh_key in require_human:      return "high"   # 规则2
    if ft == "relationship" or _is_minor_entity(cand) or (ft == "event"):  # 规则3
        return "medium"
    if ft in LOW_TYPES:                         return "low"    # 规则4
    return "medium"                                              # 规则5
```
其中 `_event_rh_key` 把"角色死亡 event"映射到 `character_death`、"伏笔回收 event"映射到 `foreshadow_payoff`,命中 require_human 即抬 `high`。

**小例子**:
- `power_system`("林枫突破筑基")→ 规则1 → `high`(auto 模式仍强制人审)。
- `knowledge`("王梅得知林枫真身")→ 规则2 命中 `knowledge_edge_change`(R10)→ `high`。
- `relationship`("林枫与赵六结为兄弟")→ 规则3 → `medium`。
- `style`("本章基调:悲怆")→ 规则4 → `low`(满足 `auto_promote_threshold` 即可自动晋升)。

### 11.4 `reconcile_drafted_state(draft, as_of_N)` — 草稿状态可达性兜底

> 第 4 节 4.4 给出签名与外壳;本节补 `as_transition` / `can_apply` / `apply` 的实现级规范。Check 阶段事后兜底:草稿最终状态必须能从 `get_world_state(N)`(第 4 节 4.4)经一串合法 transition 到达,否则报 drift 及断点。**纯确定性,无 LLM。**

**用途**:把抽取出的草稿断言转成候选状态迁移算子序列,逐个验证其在 as-of 世界状态下是否合法;任何"凭空跳变"暴露为 `STATE_UNREACHABLE`(critical),并指出断点 claim。

**输入**:`draft_claims`(第 4 节 4.2 的 `ContinuityClaim` 列表)、`world_prev = get_world_state(as_of_chapter=N-1)`、`conn`。
**输出**:`list[Issue]`(第 4 节统一 `Issue` 结构;`code='STATE_UNREACHABLE'`,`evidence_refs` 指向断点 claim 与对应 `*_log` 行)。

**`as_transition(claim) -> Transition`**:把一条 claim 翻成"状态迁移算子"。

```python
class Transition(BaseModel):
    dim: Literal["power", "knowledge", "item", "numeric", "gimmick"]   # 作用维度
    entity: str                          # 规范名(已过别名归一,第 4 节 4.3.0)
    op: str                              # power: 'rank_to'; knowledge: 'learn';
                                         # item: 'gain'|'lose'; numeric: 'set'|'delta'; gimmick: 'use'
    args: dict                           # 如 {'rank_label':'筑基期','rank_order':20}
    chapter: int
    reason_tag: Optional[str] = None     # 'breakthrough'|'injury_drop'|'item_aid'|'planted'…

def as_transition(c: "ContinuityClaim") -> Optional[Transition]:
    if c.ctype == "power_change":
        return Transition(dim="power", entity=norm(c.subject_entity), op="rank_to",
                          args={"rank_label": c.payload["rank"]}, chapter=c.chapter,
                          reason_tag=c.payload.get("reason"))
    if c.ctype == "knowledge":
        return Transition(dim="knowledge", entity=norm(c.subject_entity), op="learn",
                          args={"info_key": c.payload["info_key"]}, chapter=c.chapter)
    # … item / numeric / gimmick 同理
    return None      # 非状态型 claim(纯叙事/风格)不产生 transition
```

**`can_apply(state, transition) -> (bool, str)`**:判定迁移在当前 `state`(可变 WorldState 副本)下是否合法。核心规则按维度:

```python
def can_apply(state: WorldState, t: Transition) -> tuple[bool, str]:
    if t.dim == "power":
        ro = state.rank_order_map                  # power_ranks 全序(第 2 节 2.5.2)
        cur = state.power_order(t.entity)          # 当前 rank_order,初始可 None
        nxt = ro[t.args["rank_label"]]
        if cur is None:
            return True, "init"
        if nxt > cur:                              # 上升:不得越级,除非有合法外挂
            step = nxt - cur
            if step <= state.config.max_jump_per_step * state.config.rank_step:
                return True, "legal_rise"
            if has_breakthrough_aid(state, t.entity, t.chapter):   # 丹药/外挂/顿悟标注
                return True, "aided_rise"
            return False, f"POWER_LEVEL_SKIP step={step}"
        if nxt < cur:                              # 下降:仅 injury_drop/seal 等标注的合法下降
            if t.reason_tag in ("injury_drop", "seal"):
                return True, "legal_drop"
            return False, "ILLEGAL_POWER_DROP"
        return True, "no_change"
    if t.dim == "knowledge":
        return True, "monotone_learn"             # 知情只增(获知合法);越权"使用"由 validator 管
    if t.dim == "item":
        if t.op == "lose" and state.item_qty(t.entity, t.args["item"]) <= 0:
            return False, "ITEM_DOUBLE_SPEND"     # 没有却失去 = 双花
        return True, "ok"
    if t.dim == "numeric":
        return _numeric_can_apply(state, t)       # 单调/守恒类数值约束
    return True, "ok"
```

**主体**(对齐第 4 节 4.4 外壳):

```python
class ReconcileConfig(BaseModel):
    max_jump_per_step: int = 1       # 境界单步最多升几级
    rank_step: int = 10              # 相邻 rank_order 间距(炼气=10,筑基=20…)

def reconcile_drafted_state(draft_claims, world_prev: WorldState, conn) -> list[Issue]:
    expected = world_prev.copy()                      # 可变副本
    issues = []
    for c in topo_order(draft_claims):                # 按因果/时间序(第 4 节 4.4)
        t = as_transition(c)
        if t is None:
            continue
        ok, why = can_apply(expected, t)
        if not ok:
            issues.append(Issue(code="STATE_UNREACHABLE", severity="critical", kind="hard",
                claim_id=c.claim_id, chapter=c.chapter,
                message=(f"草稿迁移『{t.dim}:{t.op}→{t.args}』无法由 as-of(第{world_prev.as_of}章)"
                         f"经合法迁移到达({why})"),
                evidence_refs=[c.claim_id]))          # 断点 claim
        else:
            expected.apply(t)                         # 合法则推进,供后续 claim 在新状态上判定
    return issues
```

**小例子**:`as-of(40)` 林枫=筑基(order=20)。草稿 ch41 一条 `power_change` claim "林枫→元婴(order=40)"。`as_transition` 产 `Transition(dim='power', op='rank_to', args={'rank_label':'元婴期'})`。`can_apply`:`step = 40-20 = 20 > max_jump_per_step*rank_step = 1*10 = 10`,且 `has_breakthrough_aid` 查无丹药/外挂佐证 → `(False, "POWER_LEVEL_SKIP step=20")` → 产 `STATE_UNREACHABLE(critical)`,断点指向该 claim。与第 4 节 4.3.1 的 `validate_power_monotonicity` 双路印证,杜绝漏报。

### 11.5 近邻去重 `dedup(candidate)`(MVP2)

> 第 6 节召回侧的写入伴随机制;MVP2 引入。目标:避免同一事实被反复抽成多条候选,导致 staging 噪声与重复晋升。`store | merge | conflict` 三态结果回写 `fact_candidates`。

**用途**:候选入 staging 前,先在已有候选/canon 中找近邻;高于相似阈值的交 LLM 仲裁,决定独立入库、合并、还是标记潜在冲突。

**流程**:
1. **FTS 近邻**(R6):用候选的关键 token 查 `facts_fts`(canon 可检索文本)与 `drafts_fts`(L0 草稿正文),取 BM25 top-1。若 BM25 top1 与 top2 分差显著(`bm25_gap >= bm25_gap_min`),视为强候选近邻。
2. **embedding 近邻**(可选,需 `config.embedding.enabled=true`):对候选场景文本 embedding,在 `scene_vec`(第 6 节 6.2,L2 场景块向量)取最近邻,余弦相似度 `cos`。
3. **阈值判定**:`cos >= cos_threshold`(默认 0.86)**或** FTS BM25 top1 显著 → 进入仲裁;否则直接 `store`。
4. **LLM 仲裁**(模型用 `config.models.dedup` = Haiku,硬原则 10):输入候选 + 近邻,输出 `{store | merge | conflict}`。
   - `store`:两者实为不同事实 → 独立入库,候选 `status='proposed'`。
   - `merge`:同一事实的不同表述 → **补充 `evidence_refs`(并集)并保留更具体表述**(token 更多/数值更精确者为主),被并入者 `status='superseded'`(R3),主候选保留。
   - `conflict`:同一事实但值矛盾 → 候选照常 `proposed`,但回写 `conflict_flags`,交 §11.1 `detect_conflict` 与 `PromotionPolicy` 处理(很可能 `REVIEW`)。

```python
class DedupConfig(BaseModel):
    cos_threshold: float = 0.86       # embedding 余弦近邻阈
    bm25_gap_min: float = 2.0         # BM25 top1 与 top2 分差"显著"门槛
    use_embedding: bool = False       # 跟随 config.embedding.enabled
    arbiter_model: str = "claude-haiku-4-5"   # config.models.dedup

def dedup(cand, conn, cfg=DedupConfig()) -> Literal["store", "merge", "conflict"]:
    fts_hit = fts_top1(conn, cand)                          # 查 facts_fts + drafts_fts(R6)
    vec_hit = (scene_vec_nn(conn, cand) if (cfg.use_embedding) else None)  # scene_vec
    near = _pick_nearest(fts_hit, vec_hit)
    triggered = (vec_hit and vec_hit.cos >= cfg.cos_threshold) or \
                (fts_hit and fts_hit.bm25_gap >= cfg.bm25_gap_min)
    if not near or not triggered:
        return "store"
    verdict = llm_arbitrate(cand, near, model=cfg.arbiter_model)   # {store|merge|conflict}
    if verdict == "merge":
        _merge_into(near, cand, conn)     # 并 evidence_refs + 取更具体表述;被并入者 superseded
    elif verdict == "conflict":
        _flag_conflict(cand, near, conn)  # 回写 conflict_flags,留待 detect_conflict
    return verdict
```

**小例子**:已有候选 A "林枫的剑名为『青霜』"。新候选 B "林枫佩剑曰青霜剑"。BM25 top1 命中 A 且分差显著 → 仲裁;Haiku 判 `merge` → A 保留(表述更规范),B 的 `evidence_refs`(指向新章行区间)并入 A,B `status='superseded'`。staging 中只剩 A 一条,带两处出处。

### 11.6 多卷/分支隔离(MVP3):`branch_id` + copy-on-write

> MVP3 引入"平行卷/分支"(如试写不同走向)。核心:给 `facts` 与所有 `*_log` 增 `branch_id` 维度,as-of 投影签名扩 `branch_id`,分支创建用 copy-on-write 只记差异。

**用途**:在同一 `novel.db` 内隔离多条剧情分支的 World State 与 canon,互不污染,且支持跨卷续接 replay。

**表改动**(第 2 节 DDL 的增量,所有改动向后兼容:默认 `branch_id='main'`):

```sql
-- 1) 分支注册表(新增)
CREATE TABLE IF NOT EXISTS branches (
    branch_id      TEXT PRIMARY KEY,          -- 'main' / 'branch_xxx'
    parent_branch  TEXT,                       -- copy-on-write 父分支(main 为 NULL)
    fork_chapter   INTEGER NOT NULL,           -- 从父分支第几章末分叉(继承 <= 此章的状态)
    status         TEXT NOT NULL DEFAULT 'active'
                       CHECK(status IN ('active','merged','abandoned')),
    created_at     TEXT NOT NULL DEFAULT (datetime('now')),
    FOREIGN KEY(parent_branch) REFERENCES branches(branch_id)
);

-- 2) facts 与所有 *_log 增列(facts / fact_revisions / fact_candidates / promotion_log /
--    character_power_log / knowledge_edges / item_log / numeric_facts / timeline_events …)
ALTER TABLE facts                ADD COLUMN branch_id TEXT NOT NULL DEFAULT 'main';
ALTER TABLE character_power_log  ADD COLUMN branch_id TEXT NOT NULL DEFAULT 'main';
ALTER TABLE knowledge_edges      ADD COLUMN branch_id TEXT NOT NULL DEFAULT 'main';
-- … 其余 *_log 同此模式
CREATE INDEX IF NOT EXISTS idx_facts_branch ON facts(branch_id, status);
CREATE INDEX IF NOT EXISTS idx_cpl_branch   ON character_power_log(branch_id, entity_id, change_chapter);
```

**copy-on-write 语义**:分支创建**不复制**父分支历史,只在 `branches` 写一条 `(parent_branch, fork_chapter)` 差异指针;之后该分支只写**自己的新增** `*_log` 行(`branch_id='branch_xxx'`)。读取时按"继承链 + fork_chapter 边界"合并:

```python
def branch_chain(branch_id, conn) -> list[tuple[str, int]]:
    """返回 [(branch_id, upper_chapter_bound), …] 从当前分支回溯到 main。"""
    chain, bid, ub = [], branch_id, 10**9
    while bid:
        row = q_branch(conn, bid)
        chain.append((bid, ub))
        ub  = row.fork_chapter            # 父分支只继承到 fork_chapter 为止
        bid = row.parent_branch
    return chain                          # 子在前、父在后(子覆盖父)
```

**投影改动**(扩签名,第 4 节 4.4):

```python
def get_world_state(branch_id: str, as_of_chapter: int, conn) -> WorldState:
    """沿 branch_chain 合并 replay:本分支 <= as_of 的行,父分支 <= 其 fork_chapter 的行。"""
    chain = branch_chain(branch_id, conn)
    return WorldState(
        as_of=as_of_chapter, branch_id=branch_id,
        rank_order_map=load_rank_order(conn),
        _power = replay_power_branched(conn, chain, as_of_chapter),
        _knowledge=replay_knowledge_branched(conn, chain, as_of_chapter),
        # … 其余维度同理,各 replay 内按 chain 的 (branch_id, upper_bound) 取 union
    )
```
分支 replay 的 SQL 骨架(以境界为例):

```sql
-- :branch_id 当前分支, :as_of 目标章; :parent / :fork 来自 branch_chain
SELECT * FROM character_power_log
WHERE (branch_id = :branch_id AND change_chapter <= :as_of)
   OR (branch_id = :parent    AND change_chapter <= :fork)     -- 继承父分支至分叉点
ORDER BY change_chapter, created_at;                            -- 子分支同章行覆盖父
```

**跨卷续接 replay 边界**:卷 = 分支的一段。续接卷 N+1 时,以卷 N 末章 = `fork_chapter` 创建子分支(或继续 `main`);`replay` 永远到 `as_of_chapter` 为止,卷边界只是 `fork_chapter` 的特例,逻辑统一。`get_world_state(branch_id='main', as_of_chapter=卷N末)` 即得续接起点世界状态。

**向后兼容**:MVP1/2 全部隐式 `branch_id='main'`,旧调用 `get_world_state(as_of_chapter=N)` 等价 `get_world_state('main', N)`(默认参数),无需改动既有逻辑。

**小例子**:`main` 写到第 50 章。作者在第 50 章末分叉试写黑化线:`branches(branch_id='branch_dark', parent_branch='main', fork_chapter=50)`。`branch_dark` 第 51 章写"林枫黑化"只在 `branch_dark` 落 `*_log`。`get_world_state('branch_dark', 51)`:第 51 章硬状态取 `branch_dark`,第 50 章及以前继承 `main`;`main` 自身不受影响,可继续写正常线第 51 章。

### 11.7 并发模型:单写者 + WAL + 乐观锁 + 原子晋升事务

> 第 7 节主循环 Commit 阶段与第 8 节工程落地(WAL/busy_timeout)的并发权威实现。原则:**所有写经唯一队列串行化,晋升一个事务原子完成**(硬原则 9/11)。

**用途**:在"边写边校验边检索"的连写形态下,保证 canon 账本无并发损坏、晋升不出现半成品状态。

**机制**:
1. **单写者(PipelineManager 串行写队列)**:进程内唯一写线程消费写队列;**API 写与人工编辑都进同一队列**(不绕过)。读可并发(WAL 允许多读 + 单写)。
2. **WAL + busy_timeout**:连接 pragma(第 8 节 `db/connection.py`):
   ```python
   conn.execute("PRAGMA journal_mode=WAL;")     # 多读 + 单写
   conn.execute("PRAGMA busy_timeout=5000;")    # 默认 5s:锁等待而非立刻 SQLITE_BUSY
   conn.execute("PRAGMA foreign_keys=ON;")
   ```
3. **乐观锁(`version` 列)**:`facts` 增 `version INTEGER NOT NULL DEFAULT 0`。每次更新游标 `WHERE id=? AND version=:read_version`,`SET version=version+1`;`rowcount=0` 即版本冲突(有人先改了),重读重试。
   ```sql
   ALTER TABLE facts ADD COLUMN version INTEGER NOT NULL DEFAULT 0;
   ```

**候选晋升 = 单事务原子完成**(R3:`status` 终态 `promoted`;R7:`promotion_log` 在第 2 节):四步要么全成、要么全回滚。

```python
def commit_canon(cand, conn) -> str:
    """原子晋升:插 fact_revisions + 改 facts 游标(乐观锁) + 写 promotion_log + 候选→promoted。"""
    with conn:                                   # BEGIN IMMEDIATE … COMMIT/ROLLBACK(单事务边界)
        prop = BibleChangeProposal.parse_raw(cand.proposal_json)   # R8
        if prop.op == "add":
            fact_id = new_fact_id()              # fact_xxx(R2)
            rev_id  = _insert_revision(conn, fact_id, revision_no=1, op="add",
                                       new_status="canon", cand=cand)   # ① fact_revisions
            _insert_fact(conn, fact_id, prop, current_revision_id=rev_id,
                         status="canon", version=0)                     # ② facts 游标(新建)
        else:                                    # update / deprecate / retcon
            fact_id = cand.target_fact_id
            cur = _read_fact(conn, fact_id)      # 读出当前 version(乐观锁基线)
            rev_id = _insert_revision(conn, fact_id, revision_no=cur.revision_no + 1,
                                      op=prop.op, old_object=cur.object,
                                      new_object=prop.new["object"], new_status="canon",
                                      cand=cand)                         # ① fact_revisions
            n = conn.execute(                    # ② 改 facts 游标 + 乐观锁
                "UPDATE facts SET current_revision_id=?, object=?, status='canon', "
                "  valid_from_chapter=?, version=version+1, updated_at=datetime('now') "
                "WHERE id=? AND version=?",
                (rev_id, prop.new["object"], prop.valid_from_chapter, fact_id, cur.version)
            ).rowcount
            if n == 0:
                raise OptimisticLockError(fact_id)   # 触发重试(下方)
        _insert_promotion_log(conn, candidate_id=cand.candidate_id, fact_id=fact_id,
                              decision="commit_canon", route="COMMIT")   # ③ promotion_log(R7/R5)
        conn.execute("UPDATE fact_candidates SET status='promoted', "    # ④ 候选→promoted(R3)
                     "committed_revision_id=?, decided_at=datetime('now') "
                     "WHERE candidate_id=?", (rev_id, cand.candidate_id))
    return fact_id
```

**事务边界**:`with conn:`(`BEGIN IMMEDIATE` 起、`COMMIT` 止;异常 `ROLLBACK`)。四步 ①②③④ 在同一事务,杜绝"写了 revision 没改游标"或"改了游标没留痕"的半成品。

**重试策略**(乐观锁冲突 / `SQLITE_BUSY`):

```python
class WriteRetryConfig(BaseModel):
    max_retries: int = 3
    base_backoff_ms: int = 50        # 指数退避 50/100/200ms
    busy_timeout_ms: int = 5000

def with_retry(fn, cfg=WriteRetryConfig()):
    for attempt in range(cfg.max_retries + 1):
        try:
            return fn()
        except (OptimisticLockError, sqlite3.OperationalError) as e:
            if attempt == cfg.max_retries:
                raise
            sleep_ms(cfg.base_backoff_ms * (2 ** attempt))
            # 重读最新 version 后重建 cand 上下文再试(乐观锁:read-modify-write 重放)
```
因为只有**单写者**,乐观锁冲突几乎只来自"人工编辑队列项 vs 抽取晋升项"对同一 fact 的竞争;退避重读即可收敛。`busy_timeout` 兜底跨进程(如在线 `.backup`)的短锁。

**小例子**:抽取流要把 `fact_pw_lin` 从筑基改元婴(读到 `version=3`),同时人工编辑把它改回炼气也读到 `version=3`。串行队列里人工项先执行:`UPDATE … WHERE id='fact_pw_lin' AND version=3` 成功,`version→4`。抽取项执行时 `WHERE version=3` `rowcount=0` → `OptimisticLockError` → 重读得 `version=4` 与新值,重跑 `detect_conflict`(此时与"炼气"冲突)→ 很可能路由 `REVIEW`,而非盲目覆盖。无损坏、无丢更新。

### 11.8 embedding 迁移运行手册

> 第 6 节 6.2/6.4 的 `scene_vec` 与 `config.embedding`;升级向量模型时的零停机迁移与回退流程。`embedding_version` 记入 `meta_kv`(第 2 节 2.1)。

**用途**:在线把 L2 场景向量从旧模型迁到新模型(维度/语义空间不同,不可混用同一 `vec0` 表,第 2 节 2.1),保证迁移期检索不中断、可一键回退。

**前置**:`scene_vec` 行记 `embedding_model`/`embedding_dim`/`embedding_version`/`indexed_at`(第 2 节 2.1);`meta_kv` 有 `default_embedding_model`/`default_embedding_version`/`active_scene_vec_table`(active 指针)。

**步骤**:

1. **新建表**:为新模型建独立向量表 `scene_vec_v{n}`(维度按新模型,绝不复用旧表):
   ```sql
   CREATE VIRTUAL TABLE IF NOT EXISTS scene_vec_v2 USING vec0(
       scene_id TEXT PRIMARY KEY,
       embedding FLOAT[1024]        -- 新模型维度
   );
   ```
2. **后台双跑回填**:后台 worker 对所有 L2 场景块用新模型重算向量写入 `scene_vec_v2`,逐行回填 `indexed_at`(可监控进度 = 已回填行数 / L2 总数);此期间线上检索仍走旧表(`active` 指针不动)。低优先级、限速,不抢主循环资源。
   ```sql
   -- 进度查询
   SELECT COUNT(*) FILTER (WHERE indexed_at IS NOT NULL) AS done,
          (SELECT COUNT(*) FROM l2_scenes) AS total
   FROM scene_vec_v2;
   ```
3. **抽样校验召回**:回填完成后,取一组黄金查询(已知应召回的相似桥段),分别对旧表与 `scene_vec_v2` 跑召回,比较 recall@k / 排序是否不劣于旧表(默认 `recall_at_k >= 0.9 * old_recall_at_k` 才放行)。不达标则不切,排查模型/维度/分块。
4. **切 active 指针**(原子,单事务):
   ```sql
   UPDATE meta_kv SET value='scene_vec_v2' WHERE key='active_scene_vec_table';
   UPDATE meta_kv SET value=:new_model      WHERE key='default_embedding_model';
   UPDATE meta_kv SET value=:new_version    WHERE key='default_embedding_version';
   ```
   检索层始终读 `active_scene_vec_table` 指针,切换即对全线生效,无需改代码。
5. **保留旧表回退窗口**:旧表 `scene_vec_v{n-1}` **保留**默认 `retain_days=14` 天;期间任何召回异常可一键回退:把 `active_scene_vec_table` 指回旧表即可(秒级、无重算)。
6. **记版本 + 收尾**:`meta_kv.embedding_version` 记下新版本与切换时间;回退窗口过、确认稳定后,`DROP TABLE scene_vec_v{n-1}` 释放空间。

**默认参数**:

```python
class EmbeddingMigrationConfig(BaseModel):
    backfill_batch: int = 256          # 每批回填场景数
    backfill_rate_limit_qps: int = 20  # 后台限速,不抢主循环
    recall_pass_ratio: float = 0.9     # 抽样召回须 >= 旧表的 0.9 倍
    retain_days: int = 14              # 旧表回退窗口
```

**回退手册**(切后发现召回劣化):
1. 立即 `UPDATE meta_kv SET value='scene_vec_v{n-1}' WHERE key='active_scene_vec_table';`(秒级生效)。
2. 把 `default_embedding_model`/`default_embedding_version`/`embedding_version` 一并回写旧值(保持 `meta_kv` 自洽)。
3. 保留 `scene_vec_v{n}` 待排查,不立刻 DROP。
4. 因为 `embedding.enabled` 默认 `false`(第 8 节)且 embedding 仅服务 L0 草稿"查相似桥段"(非起草主力),迁移/回退**不影响 canon、不影响硬状态校验**,风险被结构性隔离。

**小例子**:从 `bge-small-zh`(512 维)升 `bge-m3`(1024 维)。建 `scene_vec_v2`→后台双跑回填 2 万个 L2 场景(进度 100%)→黄金集 recall@20 旧=0.78、新=0.81(>0.9*0.78=0.70,放行)→单事务切 `active_scene_vec_table='scene_vec_v2'` 并记 `embedding_version='bge-m3@2026-06'`→保留 `scene_vec_v1` 14 天→第 15 天 DROP。全程检索不中断。
